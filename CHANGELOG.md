## [Unreleased]

### Added

- `# bins:` `%` marker convention in scope `.nix` files (`.assets/lib/nx_doctor.sh:_check_scope_binaries` + `_check_scope_bins_in_profile`): a trailing `%` on a bin name (e.g. `# bins: fnm node% npm%`) declares "this binary is reachable on PATH but lives outside `~/.nix-profile/bin/` because a manager hook installs it elsewhere". `scope_binaries` strips the marker and still verifies via `command -v` (catches missing binary regardless of where it lives); `scope_bins_in_profile` skips marked entries (avoids false-positive FAILs for manager-installed runtimes). Three-tier convention now: unmarked = strict (both checks), `%`-marked = loose (PATH only), `(external-installer)` sentinel = skip both. The `%` token has no shell-meta meaning, so it survives the existing unquoted `for _bin in $_bins` parser without glob-expansion concerns.
- `nx setup --remove <TAB>` now completes from currently-installed scopes (read from `~/.config/nix-env/config.nix` + `local_*.nix` overlays), not just static flag names. The surface JSON already declared `value_completer: all_scopes` for `--remove`; `tests/hooks/gen_nx_completions.py` was extended to honor `value_completer` on verb-level flags (it previously only consumed the field for subcommand flags). Same fix unblocks `nx setup --omp-theme <TAB>` and `nx setup --starship-theme <TAB>` (theme-name completion). Covers bash, zsh, and PowerShell. New `tests/bats/test_completions.bats` suite (7 tests) guards the dispatch.
- `nix/configure/python_remove.sh` (new): cleanup hook for `nx scope remove python`. Mirrors `conda_remove.sh` and `nodejs_remove.sh` exactly - enumerates uv-managed state across `$UV_CACHE_DIR` (default `~/.cache/uv`), `$UV_TOOL_DIR` (default `~/.local/share/uv/tools`), `$UV_PYTHON_INSTALL_DIR` (default `~/.local/share/uv/python`), prompts `[y/N]` (suppressed under `--unattended`, auto-skipped when stdin isn't a tty), then deletes only the dirs that exist. Project-local `.venv/` directories are explicitly out of scope. Wired into `phase_scopes_apply_removes` next to the conda/nodejs arms. New `tests/bats/test_python_remove.bats` covers no-op, partial-state, enumeration, prompt-skip, and all three env-var overrides (9 tests).
- `phase_bootstrap_print_banner` (new, `nix/lib/phases/bootstrap.sh`): one-line provenance banner at the start of `nix/setup.sh` showing `Running setup from <path> (<branch> [<NIX_ENV_VERSION>])`. Format mirrors what `nx_lifecycle.sh:_nx_lifecycle_setup` already prints when launching setup from the nx CLI; the version field is appended so it matches `nx version` output. Falls back to `(<version>)` (no branch parens) on tarball installs where SCRIPT_ROOT isn't a git work tree. Always prints (no `--quiet-summary` gate). Two new tests in `test_nix_setup.bats` cover the git-repo and non-git-tree branches.
- `nx scope add <name>` now hard-rejects names that collide with managed scopes (entries in `.assets/lib/scopes.json:valid_scopes`). Prevents the silent-shadow bug where `nx scope add python` created `~/.config/nix-env/local/scopes/python.nix` + `local_python` in `config.nix`, leaving the canonical `nix/scopes/python.nix` competing with the overlay on the next setup. Implementation in `.assets/lib/nx_scope.sh`'s `add | create` arm uses `jq -e --arg n "$name" '.valid_scopes | index($n)'` against `scopes.json` (resolved via `_nx_find_lib`). Hyphen-normalized names (`k8s-base` â†’ `k8s_base`) are validated post-normalization. Three new tests in `test_overlay.bats` cover collision rejection, hyphenated collision rejection, and selectivity (non-managed names still succeed).
- `phase_bootstrap_sync_env_dir` (`nix/lib/phases/bootstrap.sh`) now copies `.assets/lib/scopes.json` into `~/.config/nix-env/` so nx commands can resolve the canonical valid_scopes list at runtime via `_nx_find_lib scopes.json`. Required by the new `nx scope add` validation; small enough that a plain `cp` on every setup is cheap.

### Changed

- `nix/scopes/nodejs.nix`: `# bins:` widened from `fnm` to `fnm node% npm%`. Restores `nx doctor` visibility for the runtime - if `fnm install --lts` never ran or the user nuked `~/.local/share/fnm/`, `scope_binaries` now WARNs with `nodejs/node` / `nodejs/npm` instead of silently passing. `scope_bins_in_profile` correctly continues to skip them (the `%` marker excludes them from the strict nix-profile check).
- `nix/scopes/terraform.nix`: `# bins:` reshaped from `terraform tfswitch tflint` (which always FAILed `scope_bins_in_profile` with `terraform/terraform` because terraform lives under `~/.local/bin/`, never `~/.nix-profile/bin/`) to `tfswitch tflint terraform%`. Fixes the FAIL the user observed; now `tfswitch` and `tflint` are audited strictly (manager binaries from nix), terraform is audited loosely (downloaded by `tfswitch --bin ~/.local/bin/terraform --latest`).
- `nix/configure/nodejs.sh`: hardened post-install with two npm config guards. (1) Pins `npm config set cafile $HOME/.config/certs/ca-bundle.crt` when the bundle exists - belt-and-suspenders complement to `NODE_EXTRA_CA_CERTS` (which only fires for shells that source the managed block; this also covers CI runners, IDE-spawned subshells, and headless `bash -c` invocations). Skipped if the user already set their own cafile. (2) Detects a stale `prefix=` setting in `~/.npmrc` left over from previous system-node setups (a common workaround for `npm install -g` EACCES) - that setting breaks fnm's PATH model since fnm only adds `~/.local/share/fnm/node-versions/<v>/installation/bin` to PATH, so globals land somewhere bash never looks. Under `--unattended`: auto-clears with `npm config delete prefix`. Interactive: prompts `[y/N]` (default no, conservative). Non-tty stdin without `--unattended`: skips the prompt and retains, mirroring the `conda_remove.sh` pattern.
- `nix/lib/phases/configure.sh:phase_configure_per_scope`: the `nodejs)` arm now forwards `${unattended:-false}` to `nodejs.sh "$1"` so the new stale-prefix guard can honor `--unattended`.
- Three regression tests in `tests/bats/test_nx_doctor.bats` covering the `%` semantics: (1) `scope_binaries` strips `%` and uses `command -v`; (2) `scope_bins_in_profile` skips `%`-marked entries; (3) a `%`-marked bin missing from PATH WARNs in `scope_binaries` while `scope_bins_in_profile` still PASSes - so manager-hook-never-ran is still caught.

## [1.6.0] - 2026-05-05

Node.js runtime management release. The `nodejs` scope now installs `fnm` (Fast Node Manager) instead of `nodejs` from nixpkgs, mirroring the `python` scope's pattern of nix installing the *manager* (uv) rather than the runtime. Prompted by `npm install -g <pkg>` failing with `EACCES: permission denied, mkdir '/nix/store/...'` because nixpkgs `nodejs` lives in the read-only Nix store. Two wins beyond unblocking global installs: per-project `.nvmrc` / `.node-version` switching now works (fnm reads them on `cd`), and the Node version is no longer pinned to whatever nixpkgs ships - users can `fnm install <version>` and `fnm use` ad-hoc. A new post-install hook bootstraps LTS automatically on first run (`fnm install --lts && fnm default lts-latest`) so `node` / `npm` work in a fresh shell immediately after `nix/setup.sh --nodejs` with no manual step. A new removal hook (`nx scope remove nodejs`) cleans up `~/.local/share/fnm/` with a confirmation prompt, mirroring `conda_remove.sh` exactly.

### Added

- `nix/configure/nodejs.sh` (new): post-install hook for the `nodejs` scope. Installs Node LTS via `fnm install --lts` only if `lts-latest` is not already present (gated on `fnm list` to avoid a network round-trip on every `nx upgrade`), then unconditionally `fnm default lts-latest` to self-heal the default symlink. Cheap and idempotent on rerun.
- `nix/configure/nodejs_remove.sh` (new): cleanup hook for `nx scope remove nodejs`. Mirrors `conda_remove.sh` exactly - enumerates installed Node versions, prompts `[y/N]` (suppressed under `--unattended`, auto-skipped when stdin isn't a tty per the `check-no-tty-read` convention), then `rm -rf "$HOME/.local/share/fnm"`. The `# :fnm` shell-init block self-strips on the next `nx profile regenerate` because it's gated on `~/.nix-profile/bin/fnm` existing.
- `# :fnm` block in the bash/zsh managed block (`.assets/lib/nx_profile.sh:_nx_render_nix_block`): emits `eval "$(fnm env --use-on-cd --shell <shell>)"`, gated on `~/.nix-profile/bin/fnm` existing. Wires node/npm onto PATH and activates per-directory version switching on `cd`.
- `#region nix:fnm` in the PowerShell profile (`.assets/config/pwsh_cfg/_aliases_nix.ps1:_NxProfileRegenerate`): pipes `fnm env --use-on-cd --shell power-shell` through `Out-String | Invoke-Expression`. Same gating, same behavior. Added to the `_NxProfileUninstall` cleanup list so `nx profile uninstall` strips it.
- Two regression tests in `tests/bats/test_nx_doctor.bats`: (1) both binary audits PASS for the new `# bins: fnm` value when fnm is in `~/.nix-profile/bin/`, and node/npm are correctly NOT reported as missing; (2) `scope_bins_in_profile` FAILs with `nodejs/fnm` when fnm has been nuked from the profile - guards against silent breakage.
- Agent guardrail in `.claude/CLAUDE.md`: explicit rule that agents must not run `make test-nix` or `make test` (Docker smoke tests; slow; user runs on demand). Agent verification stops at `make lint` + `make test-unit` + scoped bats.

### Changed

- `nix/scopes/nodejs.nix`: replaced `nodejs` with `fnm`. The `# bins:` comment now lists `fnm` (the manager that lives in `~/.nix-profile/bin/`), not `node` (the runtime that lives under `~/.local/share/fnm/node-versions/<v>/`). Mirrors `python.nix`'s `# bins: uv prek`. Doctor checks (`scope_binaries`, `scope_bins_in_profile`) audit the manager - the runtime is intentionally out of scope because it's not nix-managed. Visible effect for users with the existing `nodejs` scope: after `nix/setup.sh --upgrade`, `which node` resolves to `~/.local/share/fnm/...` instead of `~/.nix-profile/bin/node`; `npm install -g <pkg>` now succeeds.
- `nix/lib/phases/configure.sh:phase_configure_per_scope`: dispatches a new `nodejs)` arm to `nix/configure/nodejs.sh` (next to `conda)`).
- `nix/lib/phases/scopes.sh:phase_scopes_apply_removes`: dispatches a new `nodejs)` arm to `nix/configure/nodejs_remove.sh` (next to `conda)`), forwarding `${unattended:-false}`.

## [1.5.5] - 2026-05-04

Cert-flow correctness release. Three independent silent-failure modes in `phase_nix_profile_mitm_probe` conspired to skip `cert_intercept` on macOS-with-Keychain-MITM and Debian-WSL-with-`update-ca-certificates`, leaving `~/.config/certs/ca-custom.crt` missing and `NODE_EXTRA_CA_CERTS` unset for downstream Node-based pre-commit hooks (pyright via nodeenv, markdownlint, cspell). (1) The probe was gated on `ca-bundle.crt` existence, but the macOS Keychain dump always produces that file regardless of MITM. (2) System curl trusted MITM via the platform default cert store, and `curl --cacert <Mozilla-bundle>` only worked on Ubuntu - macOS Secure Transport silently ignores `--cacert`, Debian curl is additive with `SSL_CERT_FILE`. Replaced with `openssl s_client -CAfile` (same code path on every platform). (3) `cert_intercept` returned exit code 1 in the no-skips case (trailing `[ test ] && cmd` pattern), killing setup under `set -e` immediately AFTER a successful intercept. Plus a new bootstrap phase that builds `ca-bundle.crt` and self-heals stale cert env vars before any nix/git network call, so users who delete `ca-bundle.crt` to force a refresh don't hit opaque nix failures during bootstrap.

### Fixed

- `phase_nix_profile_mitm_probe` (`nix/lib/phases/nix_profile.sh`) and `build_ca_bundle` (`.assets/lib/certs.sh`): the MITM probe was gated on `ca-bundle.crt` existence, and `build_ca_bundle` short-circuited if the bundle was already present. On macOS where the Keychain dump produces `ca-bundle.crt` even without MITM (and when MITM certs are admin-installed in the Keychain, the dump *includes* them), an existing bundle silently skipped the entire probe â†’ `cert_intercept` never ran â†’ `~/.config/certs/ca-custom.crt` stayed missing â†’ the Makefile's `NODE_EXTRA_CA_CERTS := $(wildcard ...)` evaluated empty and Node-based pre-commit hooks (pyright via nodeenv, etc.) failed with `SELF_SIGNED_CERT_IN_CHAIN`. Reframed the two files as independent: `ca-bundle.crt` is now (re)built on every setup run regardless of MITM state (cheap symlink on Linux, atomic Keychain dump on macOS), and the probe is gated on `ca-custom.crt` (the cause) instead of `ca-bundle.crt` (the derivative). On MITM detection, `build_ca_bundle` runs a second time to merge the freshly intercepted `ca-custom.crt` into the macOS bundle.
- `phase_nix_profile_mitm_probe` strict probe (`nix/lib/phases/nix_profile.sh` + `_io_curl_probe_pinned` in `nix/lib/io.sh`) and `_check_cert_bundle` doctor probe (`.assets/lib/nx_doctor.sh`): the probe used system curl (nix curl is not installed by `base.nix`), which silently trusted MITM in two distinct ways: (a) on Linux/WSL, system curl trusts `/etc/ssl/certs/ca-certificates.crt` - which already contains the MITM cert because `wsl_certs_add.ps1` imported it via `update-ca-certificates`; (b) on macOS, system curl uses Apple's Secure Transport TLS backend which silently *ignores* `--cacert` and always consults the Keychain (which holds admin-installed MITM). A `curl --cacert <nix-mozilla-bundle>` attempt also failed: macOS Secure Transport ignored it entirely, and on Debian curl/OpenSSL is ADDITIVE with `--cacert` and `SSL_CERT_FILE` - both files load into the trust store, so the env-block-exported `SSL_CERT_FILE=ca-bundle.crt` (which symlinks the system store) re-added the MITM cert and the probe passed. Replaced curl entirely with `openssl s_client -CAfile <nix-mozilla-bundle> -verify_return_error`: same code path on every platform, ignores `SSL_CERT_FILE`/`SSL_CERT_DIR` when `-CAfile` is given, no Secure Transport quirk. `openssl` is already a hard dependency (in `base.nix`, used by `cert_intercept` itself).
- `cert_intercept` (`.assets/config/shell_cfg/functions.sh`): function ended with `[ $skip_count -gt 0 ] && printf ...`, leaving the test's exit code as the function's return value when no skips occurred (the common first-run case where `cert_intercept` extracts certs into a fresh `ca-custom.crt`). Caller `phase_nix_profile_mitm_probe` runs under `set -e` in `nix/setup.sh`, so the stray return-1 silently killed the whole setup at the `nix-profile` phase immediately AFTER the successful intercept - users saw "added N certificate(s)" then "Setup failed at phase 'nix-profile'". Replaced the `&&` with `if/fi` so the function always returns 0. Added a regression test in `tests/bats/test_functions.bats` (the previous bats coverage stubbed `cert_intercept` with `touch` which always returns 0, hiding the bug).
- `phase_bootstrap_ensure_certs` (new, `nix/lib/phases/bootstrap.sh`): builds `ca-bundle.crt` and self-heals stale cert env vars BEFORE any nix or git network call. Without this, a user who deletes `~/.config/certs/ca-bundle.crt` (or whose previous run left a stale path in `NIX_SSL_CERT_FILE` / git `http.sslCAInfo`) hits opaque failures during `phase_bootstrap_install_jq` because the inherited env var points at a missing file and nix's OpenSSL aborts before any of our error handlers print a message. The new phase runs after `resolve_paths` and before `detect_nix`/`verify_store`/`install_jq`. It also unsets `NIX_SSL_CERT_FILE`/`SSL_CERT_FILE` for the current run if `build_ca_bundle` couldn't (re)create the file (e.g., unsupported Linux distro without `/etc/ssl/certs`), so nix tools fall back to their bundled cacert instead of aborting.
- `_check_cert_bundle` (`.assets/lib/nx_doctor.sh`): added a network probe for the inverse-mismatch state (`ca-bundle.crt` present, `ca-custom.crt` missing, MITM still detected). Previously `nx doctor` returned PASS for this broken configuration. New check: when `ca-custom.crt` is absent, run a Mozilla-pinned probe via `openssl s_client -CAfile <nix-mozilla-bundle> -verify_return_error`; if it fails AND a `-k` retry succeeds, FAIL with a remediation pointing to `nix/setup.sh`. Skipped under `NX_DOCTOR_SKIP_NETWORK=1` (used in tests) and when nix's cacert bundle is unavailable.

## [1.5.4] - 2026-05-04

Resilience release. Three fixes triggered by a real cross-major upgrade failure (a 1.3.x install ran `nx self update` against the 1.5.x repo and ended up half-synced - old `_nx_self_sync` knew only about the pre-1.4.0 file list, so the new `nx.sh` it just installed couldn't find `nx_pkg.sh` / `nx_scope.sh` / `nx_profile.sh` / `nx_lifecycle.sh`). Now: `_nx_self_sync` delegates to `nix/setup.sh --skip-repo-update` so the **latest** `phase_bootstrap_sync_env_dir` always determines the file list (cross-major safe by construction); `nx.sh` self-heals on missing family files with a concrete recovery instruction instead of cryptic `command not found` errors; `nx version` under zsh stopped erroring with `read-only variable: status` (renamed `local status` â†’ `local ir_status` because `$status` is a zsh special read-only variable). Plus a new `check-zsh-compat` lint rule for the broader class - the existing zsh smoke test missed the bug because it ran without `install.json` and hit the early-return path before the broken `local` declaration.

### Added

- `check-zsh-compat` hook: new rule flagging `local` declarations of zsh's special read-only variables (`status`, `pipestatus`, `LINENO`, `lineno`, `argv`). Using one of these as a local name errors at runtime under zsh with `read-only variable: <name>` because zsh refuses to shadow the special. The previous test for `_nx_lifecycle_version` under zsh ran in an environment without `install.json` and exited via the early-return path, never reaching the `local status=...` declaration that breaks - this static rule catches the same class of issue at lint time without needing the runtime smoke to find it.

### Changed

- `_nx_self_sync` (`.assets/lib/nx_lifecycle.sh`) now delegates to `bash <repo>/nix/setup.sh --skip-repo-update` instead of doing its own file copy with a hardcoded list. The previous design baked the file list into the user's installed `nx_lifecycle.sh` - on a cross-major upgrade, the OLD installed sync function knew nothing about lib files added since (e.g. 1.3.x â†’ 1.5.x added `nx_pkg.sh` / `nx_scope.sh` / `nx_profile.sh` / `nx_lifecycle.sh`), so `nx self update` left the install half-synced and the new `nx.sh` couldn't load (`nx: family file nx_pkg.sh not found`). Delegating to setup.sh guarantees the *latest* `phase_bootstrap_sync_env_dir` determines the file list, so the upgrade chain stays in lockstep with whatever the repo's current code says it should be. `_nx_self_dispatch update` no longer prints the now-redundant "Run nx setup for a full environment update" follow-up since `_nx_self_sync` itself runs the full setup pipeline.

### Fixed

- `_nx_lifecycle_version` (`.assets/lib/nx_lifecycle.sh`): `nx version` errored under zsh with `read-only variable: status` whenever `install.json` was present. `$status` is a zsh special read-only variable (the exit code of the last command, equivalent to bash's `$?`), and `local status=...` errors with `read-only variable: status` because zsh can't shadow the special. Renamed the local to `ir_status`. The bug shipped in 1.5.x because the existing `nx_main version works under zsh` smoke test ran without `install.json` and hit the early-return path before reaching the broken `local` declaration; added a second smoke test that pre-seeds `install.json` so the full happy-path runs under zsh, and added a static `check-zsh-compat` rule (see Added) to catch the class of issue at lint time.
- `nx.sh` family-file source loop now self-heals on missing files. If `nx_pkg.sh` / `nx_scope.sh` / `nx_profile.sh` / `nx_lifecycle.sh` are absent (the broken-mid-upgrade state described above), the loop prints a concrete recovery instruction (`bash <repo_path>/nix/setup.sh`, looked up from `install.json:repo_path`, or a generic clone-and-setup fallback when the repo path is unknown) and replaces `nx_main` with a stub that surfaces the same message on every subsequent `nx` invocation. Previously the loop printed `nx: family file X not found` and continued, leaving the user with cryptic `command not found: _nx_<family>_<verb>` errors on the actual command. Now the user sees exactly what to run.

## [1.5.3] - 2026-05-03

Hot-fix release on top of 1.5.2 with three production fixes around `wsl.exe` invocation through `Invoke-WslExe`, two doc-rendering bugs, and release-tooling polish. The headline item is the `Process.Start` `WorkingDirectory` fix - relative paths passed to `wsl.exe` (`nix/setup.sh`, `.assets/fix/fix_no_file.sh`, ...) were resolving against the .NET CWD inherited at PowerShell startup rather than `$PWD`, silently breaking the install pipeline from any directory other than the launch dir. The test-mode `$LASTEXITCODE` fallback fix (a continuation of the 1.5.2 modularization) cleans up misleading `nix/setup.sh failed` lines on every CI run. `make release` no longer builds a local tarball (the release workflow already does it on tag push), and `build_release.sh` is now portable across Linux and macOS.

### Changed

- **`make release` (`Makefile`)**: no longer builds a local tarball before tagging. The `release.yml` workflow already builds, signs (cosign keyless via OIDC), and publishes the tarball + SBOM + checksums on tag push, so the local build was duplicate work that produced artifacts in `dist/` nothing consumed. `make release` is now validation (branch=main, clean worktree, in sync with `origin/main`, version not already tagged) â†’ confirm â†’ `git tag` â†’ `git push`. Help string updated accordingly.

### Fixed

- `Invoke-WslExe` (`modules/utils-setup/Functions/wsl_common.ps1`): `Process.Start` now sets `WorkingDirectory = $PWD.Path` explicitly. PowerShell and .NET maintain separate CWD state - `Set-Location` updates `$PWD` but not `[System.IO.Directory]::GetCurrentDirectory()`, which `Process.Start` inherits. Without this, all relative paths passed to `wsl.exe` (e.g. `nix/setup.sh`, `.assets/fix/fix_no_file.sh`) resolved against the .NET CWD from PS startup, silently targeting the wrong directory and producing `execvpe(...) failed: No such file or directory` errors.
- `docs/standards.md`: fixed two rendering bugs that truncated the Quality & Testing page mid-content. Mermaid node labels used literal newlines inside `["..."]` (invalid syntax - replaced with `<br/>`); a `</dev/tty` pattern in a table cell was misread by Python-Markdown's HTML preprocessor as a raw closing tag, silently dropping all subsequent table rows and sections.
- `Invoke-WslExe` (`modules/utils-setup/Functions/wsl_common.ps1`): test-mode fallback now resets `$global:LASTEXITCODE = 0` before invoking the mocked `wsl.exe` call. Without this, a leftover non-zero exit code from a prior real external command made `$LASTEXITCODE`-based failure checks downstream (most visibly `Install-WslScopes` after the `nix/setup.sh` call) spuriously log `<command> failed` during integration tests. Pester assertions still passed (they check call sequence, not exit codes), so the pipeline reported success - but the CI log was littered with misleading `ERROR ... nix/setup.sh failed` and `<< Failed to set up the Ubuntu WSL distro >>` lines on every run. Mocks that want to simulate failure can set `$global:LASTEXITCODE` themselves to override the default.
- `.assets/tools/build_release.sh`: checksum step now prefers `sha256sum` (coreutils, present on every Linux distro by default) and falls back to `shasum -a 256` (Perl-based, present on macOS by default). The previous hardcoded `shasum` invocation broke on Linux hosts without Perl's `Digest::SHA` script on `PATH` (most notably the user-scope WSL+Nix path used by `make release`, where `shasum` is not part of the base scope). Output format is identical between the two tools, so consumers verifying `CHECKSUMS.sha256` are unaffected.

## [1.5.2] - 2026-05-03

Modularizes `wsl/wsl_setup.ps1` (670 lines, deeply nested begin/process/clean blocks with 25+ inline `wsl.exe` calls) into 16 named, unit-testable phase functions on the existing `modules/utils-setup` PowerShell module. The orchestrator collapses to ~330 lines of named function calls; the new module surface is reusable from future WSL helper scripts. Behavior byte-identical at the production level. Manual Windows + WSL smoke-testing surfaced four real regressions the Pester suite couldn't reach (output-stream pollution, UTF-16 garble, broken `nix` progress bar, macOS Pester race) - all fixed in this release via a new `Invoke-WslExe` helper built on `[System.Diagnostics.Process]::Start` with inherited stdio. Three rounds of Copilot review folded in: silent error catches now log, restart-required signaling uses `FullyQualifiedErrorId` instead of brittle message matching, git config resists single-quote injection. The `-WebDownload` switch added to `wsl/wsl_setup.ps1` in 1.5.1 is now also propagated through `wsl/wsl_install.ps1`. Vendored ps-modules functions (`do-common`, `do-linux`, `aliases-kubectl`, `utils-install`) refreshed from upstream.

### Added

- `wsl/wsl_install.ps1`: new `-WebDownload` switch, forwarded to the `wsl/wsl_setup.ps1` invocation as `-WebDownload`. Mirrors the switch added to `wsl_setup.ps1` in 1.5.1 - lets users running the higher-level installer bypass the Microsoft Store as the distro source when the Store stalls or is policy-disabled. Existing behavior preserved when the switch is absent.
- **`Invoke-WslExe`** (`modules/utils-setup/Functions/wsl_common.ps1`): wraps `wsl.exe` via `[System.Diagnostics.Process]::Start` with `UseShellExecute = $false` and inherited std{in,out,err}. Bypasses the PowerShell pipeline entirely - wsl.exe writes directly to the terminal. Three wins: UTF-16 native output renders correctly (no PS decoder step), TTY-aware programs inside WSL detect a real terminal (nix progress bar, apt-get progress, interactive prompts), output never enters the calling function's pipeline output. `$LASTEXITCODE` is set globally to wsl.exe's exit code after `WaitForExit`. Off-Windows (Pester runners on Linux/macOS) falls back to the PS call operator so `Mock wsl.exe { ... }` can intercept; the gating uses `$IsWindows` directly to avoid env-var-based race conditions across parallel test runspaces.

### Changed

- **`wsl/wsl_setup.ps1` modularization**: full extraction of the begin / process / clean block logic into 16 named functions on the existing `modules/utils-setup` PowerShell module. Implements the design tracked in `design/wsl_setup_modularization.md`. - **wsl_setup.ps1 phase function output pollution -> array-vs-hashtable**: phase functions in `modules/utils-setup/Functions/wsl_phases.ps1` and `wsl_install.ps1` (`Invoke-WslDistroCheck`, `Invoke-WslBaseSetup`, `Install-WslScopes`, `Install-WslDistroIfMissing`, `Install-WslService`, `Invoke-WslDistroMigration`) called `wsl.exe` and helper scripts without capturing their output. Symptom: after an interactive `wsl.exe --distribution $Distro` user-profile setup, `Invoke-WslDistroCheck` returned `@('<wsl output>', <hashtable>)` instead of `<hashtable>`, and the next call (`Resolve-WslDistroScopes -Check $chk`) failed with `Cannot convert "System.Object[]" value to type "System.Collections.Hashtable"`. Caught by manual Windows + WSL smoke-testing, missed by the Pester suite (mocks returned `''` which array-property access papered over). Initial fix piped every uncaptured external command through `| Out-Default` and switched `if (-not $?)` checks to `if ($LASTEXITCODE -ne 0)`; final fix replaces native `wsl.exe` calls with `Invoke-WslExe` (Process.Start with inherited stdio - see Added). PowerShell's pipe decoded with the wrong console encoding rendered "D o w n l o a d i n g :   U b u n t u" (every other byte = NUL = space). Fixed by routing native `wsl.exe` calls through `Invoke-WslExe`, which uses `Process.Start` with inherited stdio so wsl.exe writes directly to the terminal without PowerShell's decoding step.
- **`nix` progress bar replaced with verbose log under `wsl/wsl_setup.ps1`**: piping `wsl.exe --exec nix/setup.sh @nixArgs | Out-Default` made nix's `isatty(STDOUT_FILENO)` check return 0 (the inner program saw a pipe instead of a terminal). nix fell back to a `copying path '/nix/store/...'` line per dependency instead of the live progress bar. Same TTY-detection issue affected `apt-get` progress and any other inner program that branches on isatty. Fixed by `Invoke-WslExe` preserving the parent's stdio handles - the chain `nix â†’ wsl.exe â†’ terminal` is now TTY all the way (matches pre-refactor behavior).
- **macOS Pester race under `tests/hooks/pester_parallel.ps1`**: file-level parallelism via `ForEach-Object -Parallel` runs each `*.Tests.ps1` in a separate runspace within one OS process, so `$env:*` is shared. An earlier `WSL_SETUP_TESTING` env-var-based gate in `Invoke-WslExe` raced when one suite's `AfterAll` cleared the var while the other was still running, causing fall-through to `Process.Start` on a system without `wsl.exe`. Fixed by gating on `$IsWindows` directly - the production `Process.Start` path only makes sense on Windows; off-Windows always uses the PS call operator (mockable). Verified: 139/139 tests pass under the parallel runner.
- **Silent orchestrator catches**: pre-1.5.2 `wsl_setup.ps1` had three `try/catch { exit 1 }` blocks (Install-WslDistroIfMissing, Invoke-WslDistroCheck, Invoke-WslBaseSetup) with no logging - failures left the user staring at an empty exit. Editing the user-facing message text no longer flips control flow.
- **Single-quote injection in Set-WslGitConfig**: git `user.name` / `user.email` values were interpolated into a single-quoted bash context (`git config --global user.name '$user'`). Names like `O'Connor` produced invalid bash and a theoretical injection vector. ## [1.4.0] - 2026-05-01

This release introduces the **manifest-driven nx CLI surface** - `.assets/lib/nx_surface.json` is the single source of truth for verbs, subverbs, aliases, flags, completer references, and `nx help` text. Adding a verb or flag is a one-file edit, regenerated into bash/zsh/PowerShell completers and the `nx help` body via one Python script. Four pre-commit parity hooks defend the surface against drift in any direction (completer files, PS `nx profile` dispatcher, bash `nx_main` dispatcher, lib-file installer/audit lists). Other highlights: `nix/setup.sh` auto-refreshes the source repo at start (with `--skip-repo-update` opt-out), `nx doctor` adds three new checks closing silent-failure gaps, the monolithic `nx.sh` is split into four verb-family files, and `nx_doctor.sh` is refactored into a registry pattern.

### Added

- `nix/setup.sh`: auto-refresh the source repo from upstream at start (`phase_bootstrap_refresh_repo` in `nix/lib/phases/bootstrap.sh`), mirroring `wsl/wsl_setup.ps1`'s `Update-GitRepository`. Uses the same cheap `ls-remote` pre-check (no fetch when remote tip already matches the local tracking ref). On a successful update, the script exits 0 with `repository updated to <upstream> - run the script again`. Skips silently when the working tree has uncommitted changes or HEAD has diverged from upstream (protects feature-branch dev work), when not in a git work tree (tarball install), when `git` is unavailable, or when no upstream tracking branch is configured (e.g. detached HEAD on CI).
- `nix/setup.sh --skip-repo-update`: opt-out flag for the auto-refresh, intended for repo-developer iteration ("test my uncommitted changes to setup.sh") and for callers that already refreshed the repo themselves. Wired into bash/zsh/PowerShell completers for `nx setup`. `wsl/wsl_setup.ps1` now passes it to `nix/setup.sh` since it already runs `Update-GitRepository` at script start.
- `.assets/lib/nx_surface.json`: declarative manifest for the `nx` CLI surface (verbs, subverbs, aliases, flags, dynamic completer references, summaries). Single source of truth for the completion generator, `nx help` generator, and the four parity hooks below.
- `tests/hooks/gen_nx_completions.py`: generator that emits bash/zsh/PowerShell tab completers AND the `_nx_lifecycle_help` body from `nx_surface.json`. Adding a verb or flag is a one-file edit (the manifest) instead of a four-file edit (parser + 3 completers). Side effects vs. the previous hand-written completers: zsh now offers `nx doctor` flags (`--strict`, `--json`) and `nx overlay` subverbs (`list`, `status`) which were previously missing; bash and zsh both now complete subverb aliases (e.g. `rm` for `scope remove`); bash groups subverbs sharing a completer into one elif (4 lines instead of 12 for `scope show|edit|remove|rm`).

The `_nx_lifecycle_help` body is wrapped with `# >>> nx-help generated >>>` / `# <<< nx-help generated <<<` markers and replaced in-place by the generator. Verb summaries, args representations (`<query>`, `<packages...>`, `[flags...]`), and the `(nx <verb> help)` hint for verbs with subverbs all derive from the manifest. Eliminates the verb-summary duplication (previously both heredoc and manifest had their own copies). Adds optional `help_args` schema field (used by `setup` to render `[flags...]` since its primary surface is passthrough flags, not positional args).

Catches drift in either direction - hand-editing a completer/help block, or editing the manifest without regenerating. Triggers on changes to the manifest, any generated artifact, or the generator/checker scripts.
- `check-nx-profile-parity` pre-commit hook (`tests/hooks/check_nx_profile_parity.py`): asserts the bash and PowerShell `nx profile` dispatchers expose the same subverbs. The two dispatchers are intentionally independent (they manage structurally different files - bash/zsh rc with `# >>> nix-env managed >>>` blocks vs PowerShell `$PROFILE` with `#region nix:* ... #endregion` regions) but their user-facing surface must stay in sync. Caught real drift on first run: PS dispatcher had no explicit `'help'` case (fell through to `default`); fixed by extracting the help text into `_NxProfileHelp` and dispatching both `'help'` and `default` to it, matching bash's `help | *)` pattern.

### Changed

- Removed legacy system-scoped installation layer (50+ `install_*.sh` scripts replaced by Nix scopes)
- Removed Vagrant infrastructure (hyperv, libvirt, virtualbox)
- Moved `scripts_egsave.ps1` from repo root to `.assets/scripts/scripts_egsave.ps1`
- Moved `build_release.sh` from `scripts/` to `.assets/tools/`
- Release tarball now includes `.assets/scripts/` and `.assets/check/` directories
- Unified scope/overlay UX: stripped `local_` prefix from scope names, merged overlay subcommands

### Fixed

- BSD sed grouped-command violations across all nix-path scripts
- Bash 3.2 compatibility (no mapfile, no associative arrays, no namerefs)
- Uninstaller cleanup for env-only and full removal modes
- Probe-first CA bundle handling (only build bundle when MITM proxy detected)
- `_io_nix_eval` path interpolation hardened via `builtins.getEnv` (no injection surface)
