## [1.7.1] - 2026-05-09

Lands the 16 fix commits from the first end-to-end cycle of the chunked agentic-review framework on the `test-quality` shard. v1.7.0 introduced the framework and reported the cycle's outcome as evidence the workflow held together; this patch ships the actual diffs. Every change is under `tests/` - no production code touched - and every commit references its source finding ID (`F-NNN`) so the audit trail back to `.wolf/reviews/2026-05-09-test-quality.json` stays intact post-merge. The verifier subagent confirmed all 16 as root-cause fixes (zero symptom-only, zero scope-creep, zero regression-risk); local `make lint` + `make test-unit` clean (456 bats + 139 Pester).

### Changed

- **`_run_completion` in `test_completions.bats` no longer forks `bash -c`** [F-009]. Previous helper built `COMP_WORDS` via shell-string interpolation embedded in a `bash -c` payload at quote level 3+; any apostrophe or backslash in the input would silently break the array initialization (the test would falsely pass with empty `COMP_WORDS`, hiding the very scenario being tested). Refactored to source the completer once in `setup()` and call `_nx_completions` in the parent shell with `COMP_WORDS` as positional args - eliminates the quoting bug class entirely and removes per-test subprocess overhead.
- **`_ir_flush` stable-timestamp tests replace `sleep 1.1` with a `date` function override** (`tests/bats/test_install_record.bats`) [F-006]. Previous tests relied on wall-time elapse to prove `installed_at` wasn't regenerated on a second `_ir_flush` call - cost ~2.2s per `make test-unit` run for a property a function override can verify deterministically. New approach overrides `date` with a counter file and asserts `stamp_counter == 1` after the second flush, which is strictly stronger coverage than "the timestamp value didn't change" (catches the case where `date` was called but happened to return the same value).
- **Inline `pwsh -Command` heredoc extracted to a reusable helper** (`tests/pester/_helpers/Invoke-WslSetupScenario.ps1`, new) [F-011]. The `DNS failure halts execution` test in `WslSetup.Tests.ps1` previously embedded 30+ lines of inline PowerShell that re-imported modules, redefined mocks, and ran the script - fragile escaping, couldn't share `BeforeEach` mocks, expensive to maintain. Helper takes the test scenario as a parameter; future subprocess-bound scenarios reuse the harness instead of duplicating it. (Followup `FU-001` in `wsl-orchestration` proposes the deeper fix: refactor the production DNS check to throw instead of `exit 1` so no subprocess is needed at all - deferred because it crosses shard boundaries.)
- **`zsh removed on Linux when not installed anywhere` test runs deterministically on every host** (`tests/bats/test_nix_setup.bats`) [F-015]. Previous `command -v zsh && skip` bypassed the test on developer machines that have zsh installed; production logic uses `has_system_cmd` which is already stubbed elsewhere in the file. Now stubs both `has_system_cmd` and `command -v zsh` (production calls `command -v zsh` directly), so the test exercises the same code path on CI and on dev hosts with zsh present.

### Fixed

- **Test files now exercise production code paths instead of inline reimplementations** (`tests/bats/test_nx_pkgs.bats` [F-001], `tests/bats/test_nx_scope.bats` [F-002], `tests/bats/test_nx_commands.bats` [F-010]). - **`Sync-WslSshKeys` test now asserts the WSL→Windows copy half** (`tests/pester/WslSetupPhases.Tests.ps1`) [F-005]. - **`hash -r` added in `test_pwsh_nop.bats`'s `no pwsh anywhere` test** [F-013]. Without it, bash's command-hash cache could return a stale `pwsh` path even with PATH cleared - order-dependent flakiness that wouldn't reproduce on a single-test run. - **`returns empty for empty set` Pester test renamed honestly** (`tests/pester/Scopes.Tests.ps1`) [F-004]. Body asserts `HaveCount 1` after adding `python` (PowerShell `Mandatory` validation rejects truly-empty collections); test name now reflects what's actually verified.
- **Contradictory assertion stack collapsed in `test_merge_local_certs.bats`** [F-007]. Block had `[[ "$cn1" -eq 2 ]]` followed by an explanation that the count is actually 1 followed by `[[ "$cn1" -ge 1 ]]` - the strict assertion was unreachable. Reduced to one `-ge 1` floor with one rationale sentence.
- **Vestigial `PATH="$BATS_TEST_DIRNAME:$PATH"` removed from `test_install_record.bats` teardown** [F-003]. Bats test directory contains `.bats` files (not executables) and isn't on PATH during setup; the assignment was a dead local in the teardown subshell, likely a leftover from a half-finished restore.

## [1.7.0] - 2026-05-09

The chunked agentic-review framework lands as the headline of this minor release: a manually-triggered, shard-scoped code review system that supplements the per-PR cycle with periodic structural review of the standing codebase. Per-PR review only sees the diff; this catches what's already there - drift, dead code, cross-shard inconsistencies, decisions that aged out. Nine shards with versioned charters; three subagents (reviewer / fixer / verifier) with restricted tool sets enforcing role separation; `/review` skill orchestrating five verbs (status / `<shard>` / next / act / fix); append-only mutable findings JSON; cross-cycle followups for "remember next time" items. First end-to-end cycle on the test-quality shard (lowest blast radius, picked deliberately) shipped 16 fixes; verifier confirmed all 16 root-causes; 4 cross-cycle followups captured. Plus three supporting changes: a new `check_md_html_tags` pre-commit hook (recurrence guard for the mkdocs HTML-preprocessor bug from 1.6.4); `markdownlint-cli2 --fix` enabled to auto-fix MD022/MD031/MD058 and similar structural rules; a cross-shell parity rule that activates on shell-init edits (recurrence guard for the v1.6.3 fnm self-heal incident where pwsh was forgotten). Plus a PDCA loops SVG diagram in the standards docs page, and a `design/` directory cleanup that retires the pre-framework one-off review-doc pattern.

### Added

- **Chunked agentic-review framework** (`.claude/skills/review/`, `.claude/agents/{reviewer,fixer,verifier}.md`, `design/reviews/`). Periodic, shard-scoped code review orthogonal to the per-PR cycle. Nine standing shards by concern: `certs` (cert/proxy path), `orchestration` (`nix/setup.sh` + phase library), `nx-cli` (sourced into user shells), `config-templates` (shell/pwsh/oh-my-posh), `system-installers` (root-required installers), `wsl-orchestration` (Windows-host PowerShell), `precommit-hooks`, `test-quality` (bats + Pester suites), and `enterprise-readiness` (cross-cutting posture lens). Each shard has a versioned charter under `design/reviews/charters/<shard>.md` defining: scope (file globs), "what good looks like" (narrative criteria), what NOT to flag (de-noise list), severity rubric (critical / high / medium / low), and categories (correctness / security / maintainability / testability / docs). Three subagents with restricted tool sets enforce role separation by construction: **reviewer** (`Read, Grep, Glob, Bash` only - cannot edit code; tool restriction is the bias-control mechanism), **fixer** (`Read, Edit, Write, Bash` - minimum-scope edits, one commit per finding, hard DONE marker is `make lint && make test-unit` clean - machine-verifiable, not LLM judgment), **verifier** (`Read, Grep, Bash` - independent second opinion on the fixer's diff cold, asks "did this address root cause or silence symptom?", escalates via report; cannot edit, cannot approve a PR). The `/review` skill orchestrates with five verbs dispatched from a single namespace: `status` (rotation table from `.wolf/reviews/state.json`), `<shard>` (spawn reviewer with charter + globs + open followups), `next` (auto-pick oldest from rotation state), `act <findings-path>` (triage each finding via `AskUserQuestion` apply/defer/dispute, then spawn fixer + verifier; defers/disputes accumulate in `design/reviews/accepted.md` so the next reviewer skips them), `fix <finding-id>` (single-finding retry after a verifier flag, auto-loads active findings JSON from state, prompts for revised guidance, runs fixer + verifier with `[F-NNN, retry-N]` commit suffix). Findings JSON is **append-only mutable**: original reviewer fields (`id`, `file`, `line`, `severity`, `category`, `finding`, `suggestion`) immutable; cycle agents add fields as the cycle progresses (`triage_decision`, `fixer_outcome`, `fixer_commit`, `retry_count`, `verifier_verdict`, `verifier_note`). Self-contained record per cycle. **Cross-cycle followups** persist in `.wolf/follow-ups/<home-shard>.json` (per-machine, gitignored - same lifetime convention as findings JSON): fixer emits structured followups in its final report; `/review act` writes them to the suggested home shard's file; next reviewer for that shard sees them as candidate findings; closure paths via triage (apply/defer/dispute), verifier auto-close (when the diff incidentally resolves a FU - independent bias-control check), or manual JSON edit. Cross-shard followups go to the suggested home shard's file, so a WSL refactor noticed during a test-quality fix lands in `wsl-orchestration.json`, not back in `test-quality.json`. **Manual trigger only** - no cron, no CI gate, no auto-merge. WSL distros idle-shutdown after ~8s so cron is unreliable; manual trigger also keeps the human in the loop on every cycle. Full rationale in `docs/decisions.md` "Process decisions" section; engineering-quality summary in `docs/standards.md` "Beyond automated gates: periodic review" section (with the PDCA loops SVG diagram). Charter refinement workflow documented in `design/reviews/README.md`: when triage surfaces patterns (repeated defers for the same reason, repeated severity downgrades, etc.), refine the charter from observed reality and bump the version. **First cycle on test-quality** (deliberately lowest blast radius, ~31 test files in scope) shipped 16 fixes across 16 commits in ~50 minutes of fixer time; verifier confirmed all 16 root-causes (zero symptom-only, zero scope-creep, zero regression-risk); fixer surfaced 4 cross-shard followups now living in `.wolf/follow-ups/{nx-cli,wsl-orchestration,test-quality}.json` for the next time those shards rotate. The `enterprise-readiness` shard is structurally distinct from the other eight: reviews *posture* (extension-point contracts, the maturity claims in `enterprise.md`, the org-scale benefits promised in `benefits.md`) under two binding constraints (standalone integrity must hold, no pollution by unused enterprise scaffolding); findings have three output paths instead of one (codebase fix, `docs/enterprise.md` content addition, or `design/enterprise_design.md` design-backlog entry); the `design-backlog` path records a *plan* this cycle - implementation happens in future PRs outside the review cycle.

- **`check_md_html_tags` pre-commit hook (`tests/hooks/check_md_html_tags.py`).** Forbids unbalanced tag-like `<...` substrings in `docs/*.md`. Python-Markdown's HTML preprocessor (used by mkdocs) runs BEFORE inline code-span detection, so a substring like `</dev/tty` inside backticks is still consumed as a malformed closing tag for element `dev`; the preprocessor scans forward looking for the closing `>` and silently eats subsequent table rows / headings / sections from the rendered output until it finds one. **Backticks do NOT protect against this** - the HTML pass runs first. Recurrence guard for the bug shape that silently truncated `docs/standards.md` via `` `read </dev/tty` `` in a table cell - second occurrence (prior fix recorded in 1.6.4 CHANGELOG entry; root cause + fix detail logged to `.wolf/buglog.json:bug-073` with the prevention rule for future sessions). Scoped to `docs/` only - the mkdocs-published surface; CHANGELOG and `design/` legitimately discuss the very pattern this forbids. Self-attestation escape via `<!-- md-tags-ok -->` for the rare legitimate edge case. Detects: `<` followed by alphanumeric/slash with no matching `>` on the same line, outside fenced code blocks. Accepts: balanced tags (`<div>...</div>`), autolinks (`<https://...>`, `<user@host>`), HTML entities (`&lt;`).

- **Cross-shell parity rule (`.claude/rules/cross-shell-parity.md`).** Activates on edits to bash/zsh shell-init or profile-rendering files (`.assets/lib/nx_profile.sh`, `.assets/config/shell_cfg/**`, `.assets/config/pwsh_cfg/**`, `nix/lib/phases/profiles.sh`). Forces a paired-file check: when changing one shell's behavior, identify the equivalent location on the others (bash/zsh `nx_profile.sh` ↔ pwsh `_aliases_nix.ps1` for the nix-managed profile region; per-tier alias files map likewise) and decide explicitly - mirror it, document an asymmetry inline (e.g., the `XDG_RUNTIME_DIR` self-heal correctly skips on native Windows pwsh because the var is unset there - the asymmetry IS the design and must be commented), or skip with a stated reason (e.g., bash `compdef` has no pwsh equivalent). Recurrence guard for the v1.6.3 fnm self-heal incident: the fix originally shipped for bash/zsh only; pwsh was forgotten and required a follow-up commit. Brief pointer in `.claude/CLAUDE.md` so the rule is discoverable; full paired-file map and three-outcomes flow live in the rule file itself.

- **PDCA loops SVG diagram in `docs/standards.md`** (`docs/assets/pdca-loops.svg`). Two side-by-side wheels visualizing the per-PR cycle (full PDCA - Plan / Do / Check / Act - one cycle per diff) alongside the chunked review cycle (Plan / Check / Act only - no Do, since the code already exists). Plan at 12 o'clock, clockwise to Do at 3, Check at 6, Act at 9; the right wheel's Do node and the two arcs touching it are dashed/faded to communicate "n/a." Hand-crafted SVG (mermaid is a Sugiyama DAG layout engine and produces a horizontal chain rather than a wheel for cyclic graphs - hand SVG is the honest path for the textbook PDCA visual). Uses explicit `#e8e8e8` strokes/text since SVG loaded via `<img>` can't inherit `currentColor` from the page CSS; works on the dark mkdocs Material theme.

### Changed

- **`markdownlint-cli2 --fix` mode enabled** in `.pre-commit-config.yaml`. Auto-fixes the structural-rule violations that previously required manual fix-and-re-stage cycles on every Markdown PR: MD022 (blanks around headings), MD031 (blanks around fences), MD058 (blanks around tables), MD009 (trailing whitespace), MD012 (multiple blank lines), MD038 (spaces inside code spans), MD047 (trailing newline), MD049/MD050 (emphasis/strong style), and similar normalization rules that don't require contextual judgment. Things that DO need judgment (MD040 missing fence language hint, MD041 missing top-level heading) still surface as failures for human input. ~80% of the recent Markdown fix-loops on this branch would have been auto-fixed by the new mode.

- **`design/` directory restructured** as the chunked-review framework supersedes the pre-framework one-off review-doc pattern. Five docs deleted: `review_2026-04-23.md` (one-off external snapshot, concerns shipped or addressed in `phase_1_self_contained`), `review_2026-05-02.md` (architecture review tracking 12 items, mostly shipped - the three pending items migrated to their proper homes: #6 `managed_block_drift` to `.wolf/follow-ups/nx-cli.json` as a per-machine followup, #12(a) `recommended.lock` folded into `enterprise_design.md` §8 as a fleet-coordination feature for a future minor, #11 cleanup scheduling already covered by `marker_rename_cleanup.md`), `phase_1_self_contained.md` (completed plan - all tasks shipped against v1.0+), `nix_path_review.md` (superseded by `review_2026-04-23` and resolved by `phase_1`), `implementation_plan.md` (older general plan, superseded by phase docs). `phase_2_enterprise_readiness.md` renamed to `enterprise_design.md` and de-phased: now positioned as the single design source of truth fed by the `enterprise-readiness` review shard's `design-backlog` findings, not a sequenced phase plan. `design/` post-cleanup contains four entries: `enterprise_design.md` (active design backlog), `enterprise_notes.md` (reference model for enterprise forks - consumed by `enterprise_design.md`), `marker_rename_cleanup.md` (unfinished transitional code cleanup with concrete 9-hunk plan), `reviews/` (framework artifacts: charters, accepted.md, shards.json, README). Net change: -1231 lines deleted across the five retired docs.

- **`enterprise-readiness` charter output paths split from 2 to 3.** Categories table gained a "Fix lands in" column. The previous `integration-side` category (which conflated "the user-facing doc needs new content" with "upstream needs to plan a new extension seam") split into `docs-update` (new content lands in `docs/enterprise.md` - org-side responsibility / risk / flow descriptions) and `design-backlog` (new extension-seam proposal lands in `design/enterprise_design.md` as a section in the existing template). Three explicit routing questions per finding: path 1 vs 2/3 ("does an org without this capability pay any cost?" - if yes, push to docs or design backlog), path 2 vs 3 ("is this org-side responsibility or upstream-design?" - first goes to enterprise.md, second goes to enterprise_design.md). `design-backlog` findings record a *plan* this cycle; the implementation happens in future PRs outside the review cycle - keeps the cycle bounded (~30-60 min) while letting design proposals accumulate without context-switching to implementation.

## [1.6.4] - 2026-05-09

Cross-version upgrade walk pipeline + the `--env-only` bug it surfaced. The walk -- a sequential CI workflow that installs every released tag, switches the repo to HEAD, runs `nx setup`, and verifies the result -- catches a class of regression the platform matrix can't: a refactor on HEAD that breaks the migration path from a real installed version. Locally validated against v1.5.0 → v1.6.3 (10/10 PASS); v1.1.0 still fails verify because it predates the family-file split, which sets the upgrade-supported floor at v1.2.0 (workflow + Make target default a tighter floor of v1.5.0 to match what's actually deployed). Plus a complete restructure of the Quality & Testing docs page so the same backpressure framing the project sells to developers also describes how the codebase enforces itself.

### Added

- **Cross-version upgrade walk pipeline (`.github/workflows/test_upgrade_walk.yml` + `.github/scripts/upgrade_walk.sh`).** New CI workflow that walks every released tag newest → oldest in a single sequential job, installs at the old version, switches the repo to HEAD, runs `nx setup --skip-repo-update` (the user-facing upgrade path), and verifies family files synced + `nx version` matches HEAD + `nx doctor --strict` passes. Cheap relative to the platform matrix: ONE Nix install per run, package cache warms across iterations. Only user-scope state (`~/.config/nix-env/`, managed shell blocks, alias files) is wiped between iterations via `nix/uninstall.sh --env-only`; `/nix/store` and the per-user nix profile persist (those are validated by the matrix). Triggers: `workflow_dispatch` (with optional `from-version` and `floor-version` inputs) plus a weekly cron (Mondays 06:00 UTC). Default floor is `v1.5.0` (post-modular-split era) -- bump via the workflow input as the deployed install base ages forward. The walk extracted to a shared bash script so the same code runs in CI and locally; the workflow YAML is now ~30 lines of trigger + checkout + nix install + script invocation. `WALK_FLOOR` semver filter implemented via awk numeric comparison (`p[1]*1000000 + p[2]*1000 + p[3]`), version comparison strips `git describe`'s `-N-gSHA` suffix so non-tag HEADs don't trip the version-mismatch warning. Surfaces the "upgrade-supported floor" automatically -- the oldest version that still upgrades cleanly to HEAD.
- **`make test-upgrade-walk` Makefile target.** Local Docker reproduction of the CI walk, sharing `.github/scripts/upgrade_walk.sh` so there's no drift between "what CI runs" and "what we reproduce locally". Builds a minimal `.assets/docker/Dockerfile.upgrade-walk` (ubuntu + git + curl + nix install, ~150s one-time build), then `docker run`s the walk script with the repo bind-mounted read-only and cloned to a writable scratch dir inside the container. Default `WALK_FLOOR=v1.5.0` matches the workflow YAML; override via `make test-upgrade-walk WALK_FLOOR=v1.6.0` (tighter), `WALK_VERSIONS=v1.5.5` (single tag), or `TARGET_REF=feature/foo` (test against a non-HEAD ref). MITM cert handling reuses the existing `ENSURE_ROOT_CERT` / `CLEANUP_ROOT_CERT` machinery so the target works behind corporate proxies without configuration. Slow (~20+ min for the full walk), USER ONLY -- not part of `make test`.
- **Quality & Testing docs page restructured (`docs/standards.md`).** Top-down narrative: by-the-numbers → why these standards exist (backpressure framing, citing Eledath's "Levels of Agentic Engineering") → what we defend (the four invariants: bash 3.2 compat, scope consistency, idempotency, install provenance) → pre-commit hooks (split into custom + vendored subsections, both tables rewritten to lead with WHY rather than implementation specifics) → unit tests → CI pipelines (merged section: workflow overview table + integration matrix mermaid + cross-version upgrade walk + docker smoke tests) → Make as the unified entry point (new dedicated chapter covering the one-contract-four-consumers symmetry, tab completion across bash/zsh/pwsh, MITM proxy transparency, and the agentic-backpressure multiplier). Net change: -25 lines despite adding the new framing -- the merged CI section and dropped code dump trim more than the intro adds.

### Fixed

- **`nix/uninstall.sh --env-only` no longer breaks Nix.** Surfaced while building the cross-version upgrade walk locally in Docker: after one iteration's `--env-only` cleanup, the next iteration's `nix/setup.sh` aborted with `Nix is not installed`. Step 1i in `run_phase1` was unconditionally removing `~/.nix-profile` and `~/.local/state/nix` -- the `nix` binary still existed in `/nix/store`, but the user had no PATH entry and no profile state, so they couldn't invoke it. That contradicts `--env-only`'s documented intent ("Remove nix-env state only, keep Nix installed"). Removed step 1i; the equivalent cleanup already lives in `run_phase2` for the `--all` and interactive paths, where wiping `/nix/store` and the profile together is the desired behavior. Updated `Dockerfile.test-nix`'s post-uninstall checks to assert `~/.nix-profile`, `~/.local/state/nix`, and a working `nix --version` after `--env-only` so the regression can't sneak back.

## [1.6.3] - 2026-05-09

### Added

- **`man-db` in `nix/scopes/base.nix`.** Provides `man`, `apropos`, `whatis` at user scope, with `~/.nix-profile/share/man/` already on the default `MANPATH` so nix-installed packages' man pages are discoverable out of the box. Distro-installed `man-db` only sees `/usr/share/man/`, missing every nix package; rootless containers (Coder) and Mac users without Xcode CLT have no `man` at all. Single uniform answer across install paths. Picked `man-db` over `mandoc` for full `apropos` index support.

### Fixed

- **`/run/user/$DEF_UID/` intermittently missing on WSL boot, breaking fnm and other XDG_RUNTIME_DIR consumers.** Manifests as fnm failing with `Can't create the symlink for multishells at "/run/user/<uid>/fnm_multishells/..." No such file or directory` on a shell start, but only on some boots -- a distro can run for days without hitting it, then break on a fresh `wsl --terminate <distro>` cycle. Root cause: WSL's `wsl.exe` bash entry doesn't traverse the full PAM session stack, so `pam_systemd.so session` (which would tell `logind` to create the user runtime dir) doesn't fire on shell entry. The dir normally appears as a side effect of *other* PAM-session-using actions in the same WSL boot -- `sudo`, package installs, an SSH-into-WSL session -- so users only hit the bug when they happen to invoke fnm before any other PAM-session-firing command. Debian/Ubuntu mask this more reliably (pam_systemd in `common-session`, plus `wsl-pro-service` on Store-distributed Ubuntu creates the dir directly), but the underlying PAM-skip is the same on every distro. **Fix**: `Invoke-WslBaseSetup` (in `modules/utils-setup/Functions/wsl_phases.ps1`) now composes the `[boot] command` value in `/etc/wsl.conf` to prefix `command -v systemctl >/dev/null 2>&1 && systemctl start user-runtime-dir@$DEF_UID.service 2>/dev/null;` before the existing `autoexec.sh` invocation. `user-runtime-dir@.service` is a `Type=oneshot` systemd unit (creates `/run/user/<UID>` with proper perms via `systemd-user-runtime-dir`, then exits -- no persistent process), so this doesn't spawn a long-running `systemd --user` and doesn't interfere with WSL2's idle-shutdown timer (in contrast to `loginctl enable-linger`, which would). `$DEF_UID` is read from `$Check.def_uid` (already populated by `Invoke-WslDistroCheck` from `check_distro.sh`'s parse of `/etc/wsl-distribution.conf`'s `defaultUid`, falling back to `id -u`). Also dropped the `if (-not $Check.wsl_boot)` gate that previously skipped wsl.conf updates on existing installs -- the gate prevented existing distros from picking up the new prefix on `nix/setup.sh` re-run. Pester: retired the `'skips Set-WslConf when wsl_boot is true'` test in favor of two new tests (Set-WslConf always invoked; rendered command contains `systemctl start user-runtime-dir@<def_uid>.service` and the `command -v systemctl` guard). **Defense-in-depth**: shell-init self-heal across all three shells -- the `# :fnm` block in `.assets/lib/nx_profile.sh:_nx_render_nix_block` (renders into both `~/.bashrc` and `~/.zshrc`) and the `#region nix:fnm` in `.assets/config/pwsh_cfg/_aliases_nix.ps1:_NxProfileRegenerate` (renders into `CurrentUserAllHosts`) now check `XDG_RUNTIME_DIR` at shell start; if set but the dir doesn't exist, redirect to `/tmp/runtime-$(id -u)` (mode `0700`, matching logind's perms) before invoking `fnm env`. Covers rootless containers (Coder, etc.) where `logind` isn't available at all and the wsl.conf fix doesn't apply. Native Windows pwsh has `$env:XDG_RUNTIME_DIR` unset, so the inner block is skipped and `id -u` (Linux/macOS-only) never runs there.

- **`nx setup` re-realized the buildEnv on every run on single-user nix.** Symptom: each `nx setup` invocation creates a new profile generation, GC reclaims the previous one (~175 MiB churn per run), `nix profile upgrade` reports `upgrading from path:...?lastModified=A to path:...?lastModified=B` even when the user changed nothing. Confirmed on Coder (single-user nix, no daemon) where back-to-back `nx setup` runs both took ~25s and freed exactly 175.9 MiB through gc each time, with `narHash` of the path:flake stable but `lastModified` advancing every run. Three independent writers were stamping `~/.config/nix-env/` mtimes without content changes, advancing `path:flake.lastModified` and triggering `nix profile upgrade` to re-realize the buildEnv into a new generation (same store paths logically -- just a wrapper derivation that depends on the lastModified-laden flake URL). On daemon-mode nix (typical WSL setup) this is masked by the daemon's smarter "already-installed" handling; on single-user nix it surfaces fully. Three-part fix:
  - `install_atomic` (`.assets/lib/helpers.sh`): now compares the rendered temp file against the destination via `cmp -s` and skips the `mv -f` rename when content is identical, preserving the destination's mtime. Two new tests cover the skip path and the `--upgrade`-overrides-skip path.
- **`gh auth login` device-code prompt invisible during `phase_configure_gh`.** Both `nix/configure/gh.sh` and `nix/configure/git.sh` were wrapped in `_io_run`, which captures stderr to a tempfile and replays it only on failure. `gh auth login -w` writes its device-code line (`! First copy your one-time code: XXXX-XXXX`) to stderr; `bash read -rp` does the same. Result: user saw the three stdout questions, the browser opened to the device-auth URL, but the code itself never appeared on screen -- and stdin stayed wired to the terminal, so the user was left typing blind. Both calls now invoke the script directly. Other `_io_run` callers (per-scope configure scripts) stay wrapped -- they're non-interactive by design and benefit from the structured failure replay.
- **Cross-run dedup in `cert_intercept` and `merge_local_certs`.** Both serial-extraction code paths used `openssl storeutl -noout -text -certs <bundle>` as a fast path with a PEM-block-walk fallback. The fast path silently emits `<decimal> (0x<lowercase-hex>)` per cert, while every other code path (the WSL fork's `ConvertTo-PEM` headers, `cert_intercept`'s own per-cert append, `merge_local_certs`' per-source-cert extraction) uses `openssl x509 -serial`'s uppercase-hex format. Substring dedup against the storeutl format never matched, so any second invocation re-appended every cert. Surfaced when `merge_local_certs` ran against a real ca-custom.crt populated by `wsl_certs_add.ps1`: the PG Root CA cert (serial `59F7D8B4`) ended up in the bundle twice. Both functions now use the PEM-block walk exclusively (the fallback became the only path), keeping serial format normalized end-to-end. The fast path was the load-bearing one for cross-run idempotency, but it was structurally wrong from day one.

## [1.5.5] - 2026-05-04

Cert-flow correctness release. Three independent silent-failure modes in `phase_nix_profile_mitm_probe` conspired to skip `cert_intercept` on macOS-with-Keychain-MITM and Debian-WSL-with-`update-ca-certificates`, leaving `~/.config/certs/ca-custom.crt` missing and `NODE_EXTRA_CA_CERTS` unset for downstream Node-based pre-commit hooks (pyright via nodeenv, markdownlint, cspell). (1) The probe was gated on `ca-bundle.crt` existence, but the macOS Keychain dump always produces that file regardless of MITM. (2) System curl trusted MITM via the platform default cert store, and `curl --cacert <Mozilla-bundle>` only worked on Ubuntu - macOS Secure Transport silently ignores `--cacert`, Debian curl is additive with `SSL_CERT_FILE`. Replaced with `openssl s_client -CAfile` (same code path on every platform). (3) `cert_intercept` returned exit code 1 in the no-skips case (trailing `[ test ] && cmd` pattern), killing setup under `set -e` immediately AFTER a successful intercept. Plus a new bootstrap phase that builds `ca-bundle.crt` and self-heals stale cert env vars before any nix/git network call, so users who delete `ca-bundle.crt` to force a refresh don't hit opaque nix failures during bootstrap.

### Fixed

- `phase_nix_profile_mitm_probe` (`nix/lib/phases/nix_profile.sh`) and `build_ca_bundle` (`.assets/lib/certs.sh`): the MITM probe was gated on `ca-bundle.crt` existence, and `build_ca_bundle` short-circuited if the bundle was already present. On macOS where the Keychain dump produces `ca-bundle.crt` even without MITM (and when MITM certs are admin-installed in the Keychain, the dump *includes* them), an existing bundle silently skipped the entire probe → `cert_intercept` never ran → `~/.config/certs/ca-custom.crt` stayed missing → the Makefile's `NODE_EXTRA_CA_CERTS := $(wildcard ...)` evaluated empty and Node-based pre-commit hooks (pyright via nodeenv, etc.) failed with `SELF_SIGNED_CERT_IN_CHAIN`. Reframed the two files as independent: `ca-bundle.crt` is now (re)built on every setup run regardless of MITM state (cheap symlink on Linux, atomic Keychain dump on macOS), and the probe is gated on `ca-custom.crt` (the cause) instead of `ca-bundle.crt` (the derivative). On MITM detection, `build_ca_bundle` runs a second time to merge the freshly intercepted `ca-custom.crt` into the macOS bundle.
- `phase_nix_profile_mitm_probe` strict probe (`nix/lib/phases/nix_profile.sh` + `_io_curl_probe_pinned` in `nix/lib/io.sh`) and `_check_cert_bundle` doctor probe (`.assets/lib/nx_doctor.sh`): the probe used system curl (nix curl is not installed by `base.nix`), which silently trusted MITM in two distinct ways: (a) on Linux/WSL, system curl trusts `/etc/ssl/certs/ca-certificates.crt` - which already contains the MITM cert because `wsl_certs_add.ps1` imported it via `update-ca-certificates`; (b) on macOS, system curl uses Apple's Secure Transport TLS backend which silently *ignores* `--cacert` and always consults the Keychain (which holds admin-installed MITM). A `curl --cacert <nix-mozilla-bundle>` attempt also failed: macOS Secure Transport ignored it entirely, and on Debian curl/OpenSSL is ADDITIVE with `--cacert` and `SSL_CERT_FILE` - both files load into the trust store, so the env-block-exported `SSL_CERT_FILE=ca-bundle.crt` (which symlinks the system store) re-added the MITM cert and the probe passed. Replaced curl entirely with `openssl s_client -CAfile <nix-mozilla-bundle> -verify_return_error`: same code path on every platform, ignores `SSL_CERT_FILE`/`SSL_CERT_DIR` when `-CAfile` is given, no Secure Transport quirk. `openssl` is already a hard dependency (in `base.nix`, used by `cert_intercept` itself).
- `cert_intercept` (`.assets/config/shell_cfg/functions.sh`): function ended with `[ $skip_count -gt 0 ] && printf ...`, leaving the test's exit code as the function's return value when no skips occurred (the common first-run case where `cert_intercept` extracts certs into a fresh `ca-custom.crt`). Caller `phase_nix_profile_mitm_probe` runs under `set -e` in `nix/setup.sh`, so the stray return-1 silently killed the whole setup at the `nix-profile` phase immediately AFTER the successful intercept - users saw "added N certificate(s)" then "Setup failed at phase 'nix-profile'". Replaced the `&&` with `if/fi` so the function always returns 0. Added a regression test in `tests/bats/test_functions.bats` (the previous bats coverage stubbed `cert_intercept` with `touch` which always returns 0, hiding the bug).
- `phase_bootstrap_ensure_certs` (new, `nix/lib/phases/bootstrap.sh`): builds `ca-bundle.crt` and self-heals stale cert env vars BEFORE any nix or git network call. Without this, a user who deletes `~/.config/certs/ca-bundle.crt` (or whose previous run left a stale path in `NIX_SSL_CERT_FILE` / git `http.sslCAInfo`) hits opaque failures during `phase_bootstrap_install_jq` because the inherited env var points at a missing file and nix's OpenSSL aborts before any of our error handlers print a message. The new phase runs after `resolve_paths` and before `detect_nix`/`verify_store`/`install_jq`. It also unsets `NIX_SSL_CERT_FILE`/`SSL_CERT_FILE` for the current run if `build_ca_bundle` couldn't (re)create the file (e.g., unsupported Linux distro without `/etc/ssl/certs`), so nix tools fall back to their bundled cacert instead of aborting.
- `_check_cert_bundle` (`.assets/lib/nx_doctor.sh`): added a network probe for the inverse-mismatch state (`ca-bundle.crt` present, `ca-custom.crt` missing, MITM still detected). Previously `nx doctor` returned PASS for this broken configuration. New check: when `ca-custom.crt` is absent, run a Mozilla-pinned probe via `openssl s_client -CAfile <nix-mozilla-bundle> -verify_return_error`; if it fails AND a `-k` retry succeeds, FAIL with a remediation pointing to `nix/setup.sh`. Skipped under `NX_DOCTOR_SKIP_NETWORK=1` (used in tests) and when nix's cacert bundle is unavailable.

Resilience release. ### Added

- `check-zsh-compat` hook: new rule flagging `local` declarations of zsh's special read-only variables (`status`, `pipestatus`, `LINENO`, `lineno`, `argv`). The previous test for `_nx_lifecycle_version` under zsh ran in an environment without `install.json` and exited via the early-return path, never reaching the `local status=...` declaration that breaks - this static rule catches the same class of issue at lint time without needing the runtime smoke to find it.

### Changed

- `_nx_self_sync` (`.assets/lib/nx_lifecycle.sh`) now delegates to `bash <repo>/nix/setup.sh --skip-repo-update` instead of doing its own file copy with a hardcoded list. Delegating to setup.sh guarantees the *latest* `phase_bootstrap_sync_env_dir` determines the file list, so the upgrade chain stays in lockstep with whatever the repo's current code says it should be. `_nx_self_dispatch update` no longer prints the now-redundant "Run nx setup for a full environment update" follow-up since `_nx_self_sync` itself runs the full setup pipeline.

### Fixed

- `_nx_lifecycle_version` (`.assets/lib/nx_lifecycle.sh`): `nx version` errored under zsh with `read-only variable: status` whenever `install.json` was present. `$status` is a zsh special read-only variable (the exit code of the last command, equivalent to bash's `$?`), and `local status=...` errors with `read-only variable: status` because zsh can't shadow the special. Renamed the local to `ir_status`. The bug shipped in 1.5.x because the existing `nx_main version works under zsh` smoke test ran without `install.json` and hit the early-return path before reaching the broken `local` declaration; added a second smoke test that pre-seeds `install.json` so the full happy-path runs under zsh, and added a static `check-zsh-compat` rule (see Added) to catch the class of issue at lint time.
- `nx.sh` family-file source loop now self-heals on missing files. If `nx_pkg.sh` / `nx_scope.sh` / `nx_profile.sh` / `nx_lifecycle.sh` are absent (the broken-mid-upgrade state described above), the loop prints a concrete recovery instruction (`bash <repo_path>/nix/setup.sh`, looked up from `install.json:repo_path`, or a generic clone-and-setup fallback when the repo path is unknown) and replaces `nx_main` with a stub that surfaces the same message on every subsequent `nx` invocation. Previously the loop printed `nx: family file X not found` and continued, leaving the user with cryptic `command not found: _nx_<family>_<verb>` errors on the actual command. Now the user sees exactly what to run.

Fixed by routing native `wsl.exe` calls through `Invoke-WslExe`, which uses `Process.Start` with inherited stdio so wsl.exe writes directly to the terminal without PowerShell's decoding step.
- **`nix` progress bar replaced with verbose log under `wsl/wsl_setup.ps1`**: piping `wsl.exe --exec nix/setup.sh @nixArgs | Out-Default` made nix's `isatty(STDOUT_FILENO)` check return 0 (the inner program saw a pipe instead of a terminal). nix fell back to a `copying path '/nix/store/...'` line per dependency instead of the live progress bar. Same TTY-detection issue affected `apt-get` progress and any other inner program that branches on isatty. Fixed by `Invoke-WslExe` preserving the parent's stdio handles - the chain `nix → wsl.exe → terminal` is now TTY all the way (matches pre-refactor behavior).
- **macOS Pester race under `tests/hooks/pester_parallel.ps1`**: file-level parallelism via `ForEach-Object -Parallel` runs each `*.Tests.ps1` in a separate runspace within one OS process, so `$env:*` is shared. An earlier `WSL_SETUP_TESTING` env-var-based gate in `Invoke-WslExe` raced when one suite's `AfterAll` cleared the var while the other was still running, causing fall-through to `Process.Start` on a system without `wsl.exe`. Fixed by gating on `$IsWindows` directly - the production `Process.Start` path only makes sense on Windows; off-Windows always uses the PS call operator (mockable). Verified: 139/139 tests pass under the parallel runner.
- **Silent orchestrator catches**: pre-1.5.2 `wsl_setup.ps1` had three `try/catch { exit 1 }` blocks (Install-WslDistroIfMissing, Invoke-WslDistroCheck, Invoke-WslBaseSetup) with no logging - failures left the user staring at an empty exit. Each catch now logs `<phase> failed: <message>` via `Show-LogContext` before exiting. The brittle restart-required signaling (`Exception.Message -eq 'restart required'` literal string match) is replaced with a custom `ErrorRecord` whose `FullyQualifiedErrorId = 'WslRestartRequired'`; `Install-WslService` throws the ErrorRecord, the orchestrator matches on FQErrorId. Editing the user-facing message text no longer flips control flow.
- **Single-quote injection in Set-WslGitConfig**: git `user.name` / `user.email` values were interpolated into a single-quoted bash context (`git config --global user.name '$user'`). Names like `O'Connor` produced invalid bash and a theoretical injection vector. Now escapes single quotes via the standard close+escape+reopen pattern (`$user.Replace("'", "'\''")`) before interpolation.

## [1.4.0] - 2026-05-01

This release introduces the **manifest-driven nx CLI surface** - `.assets/lib/nx_surface.json` is the single source of truth for verbs, subverbs, aliases, flags, completer references, and `nx help` text. Adding a verb or flag is a one-file edit, regenerated into bash/zsh/PowerShell completers and the `nx help` body via one Python script. Four pre-commit parity hooks defend the surface against drift in any direction (completer files, PS `nx profile` dispatcher, bash `nx_main` dispatcher, lib-file installer/audit lists). Other highlights: `nix/setup.sh` auto-refreshes the source repo at start (with `--skip-repo-update` opt-out), `nx doctor` adds three new checks closing silent-failure gaps, the monolithic `nx.sh` is split into four verb-family files, and `nx_doctor.sh` is refactored into a registry pattern.

### Added

- `nix/setup.sh`: auto-refresh the source repo from upstream at start (`phase_bootstrap_refresh_repo` in `nix/lib/phases/bootstrap.sh`), mirroring `wsl/wsl_setup.ps1`'s `Update-GitRepository`. Uses the same cheap `ls-remote` pre-check (no fetch when remote tip already matches the local tracking ref). On a successful update, the script exits 0 with `repository updated to <upstream> - run the script again`. Skips silently when the working tree has uncommitted changes or HEAD has diverged from upstream (protects feature-branch dev work), when not in a git work tree (tarball install), when `git` is unavailable, or when no upstream tracking branch is configured (e.g. detached HEAD on CI).

Wired into bash/zsh/PowerShell completers for `nx setup`. `wsl/wsl_setup.ps1` now passes it to `nix/setup.sh` since it already runs `Update-GitRepository` at script start.
- `.assets/lib/nx_surface.json`: declarative manifest for the `nx` CLI surface (verbs, subverbs, aliases, flags, dynamic completer references, summaries). Single source of truth for the completion generator, `nx help` generator, and the four parity hooks below.

The `_nx_lifecycle_help` body is wrapped with `# >>> nx-help generated >>>` / `# <<< nx-help generated <<<` markers and replaced in-place by the generator. Verb summaries, args representations (`<query>`, `<packages...>`, `[flags...]`), and the `(nx <verb> help)` hint for verbs with subverbs all derive from the manifest. Triggers on changes to the manifest, any generated artifact, or the generator/checker scripts.

The two dispatchers are intentionally independent (they manage structurally different files - bash/zsh rc with `# >>> nix-env managed >>>` blocks vs PowerShell `$PROFILE` with `#region nix:* ... #endregion` regions) but their user-facing surface must stay in sync. EOF` help heredoc), `EOF: command not found`, then a syntax error near `;;`.
