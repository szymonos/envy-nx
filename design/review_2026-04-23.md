# Brutally honest review: `envy-nx`

_Date: 2026-04-23_
_Reviewer: external code review pass_
_Scope: whole-repo assessment based on `ARCHITECTURE.md`, `docs/`, `nix/`, `.assets/`, `tests/`, and CI configuration._

## Verdict upfront

This is a **genuinely high-quality personal/small-team tool, over-marketed as
enterprise infrastructure**. The engineering craftsmanship is well above
average for this category (shell-based env bootstrappers). But the "enterprise
readiness" narrative in `docs/enterprise.md` and the README is a stretch that
a serious InfoSec / Platform review would puncture within 30 minutes.

Rating:

| Dimension                             | Score  |
| ------------------------------------- | ------ |
| Technical craftsmanship               | 8 / 10 |
| Fitness-for-purpose as a bootstrapper | 8 / 10 |
| Actual enterprise readiness           | 4 / 10 |
| Honesty of self-marketing             | 5 / 10 |

## What is genuinely strong

1. **Architecture discipline is real.** `nix/setup.sh` as a ~125-line
   orchestrator that sources 9 phase libraries (`bootstrap`, `platform`,
   `scopes`, `nix_profile`, `configure`, `profiles`, `post_install`, `summary`)
   with explicit `# Reads: / Writes:` contracts and `_io_*` wrappers for
   side-effect stubbing in tests is the kind of structure most shell codebases
   never reach. This is uncommon and commendable
   (`nix/setup.sh:41-44`, `nix/lib/phases/bootstrap.sh:1-9`).
2. **Test coverage is real**, if inflated. ~283 bats `@test` cases and ~136
   Pester `It` blocks - 419 total, matching the "412 tests" claim closely.
   That's substantial for a shell project.
3. **Bash 3.2 + BSD sed discipline** enforced by a pre-commit hook
   (`check_bash32`) is a legitimately hard constraint that most
   "cross-platform" scripts claim but violate. The `ARCHITECTURE.md:500-516`
   constraint list is honest and enforced.
4. **MITM proxy / corporate cert handling** is the standout feature.
   `NIX_SSL_CERT_FILE`, `NODE_EXTRA_CA_CERTS`, VS Code Server
   `server-env-setup`, the probe in `phase_nix_profile_mitm_probe` - this is
   the pain point most competing tools don't solve, and the rationale in
   `ARCHITECTURE.md:385-421` is the clearest thinking in the repo.
5. **Managed-block pattern** for rc file injection (bash
   `# >>> nix-env managed >>>`, PowerShell `#region nix:base`) with idempotent
   regenerate/remove is correct and durable. `nx uninstall --env-only` being
   CI-asserted to preserve generic config is a detail most tools get wrong.
6. **Design-decision documentation is exceptionally honest** where it does
   exist - the rationale for dropping `set -u` (`ARCHITECTURE.md:371-383`),
   keeping `nixpkgs-unstable` unpinned by default, rejecting
   `pip-system-certs`, the bootstrap-jq dance via `base_init.nix` - this is
   all well-reasoned.

## Where it's weaker than advertised

### 1. The "enterprise ready" framing is oversold

`docs/enterprise.md` rates itself "Strong" on 7 of 9 dimensions. Looking at
the actual substance:

- **"Fleet telemetry: Scaffold only"** - accurate, but in plain English this
  means _there is no telemetry_. An `install.json` file on each laptop is not
  telemetry; it's a log file. A real enterprise would need a collector,
  schema contract, retention policy, PII review - none of which exists.
- **"MDM integration: Ready (not included)"** - translation: _not
  integrated_. Having an `--unattended` flag is table stakes, not MDM
  integration.
- **"Policy enforcement: Extension point"** - translation: _no policy
  enforcement_. "Put it in your overlay" is pushing the entire enterprise
  layer onto the consumer.
- **"Distribution: git clone"** acknowledged (`enterprise.md:92-99`) but this
  alone disqualifies it for most regulated environments. No signed releases,
  no SBOM, no tarball pipeline ("designed but not yet implemented").
- **Single-maintainer project** with 9 commits, no releases, no tags (all
  work in unreleased master). Calling something "enterprise-ready" before
  v1.0 is questionable.

### 2. Supply-chain posture is the real blocker

The enterprise doc waves at this but understates it:

- Depends on `nixpkgs-unstable` by default - _unstable_ in the name is not a
  detail InfoSec will shrug off.
- Pipes `curl | sh` from `install.determinate.systems` during bootstrap
  (`nix/lib/phases/bootstrap.sh:85-86`). Third-party curl-pipe-sh is
  precisely what most enterprise policies prohibit.
- No checksum pinning on the installer, no offline/air-gapped install story
  beyond "use an internal mirror (not implemented)".
- No GPG-signed tags, no SLSA attestation, no SBOM output.
- `nx pin` exists (good) but it's opt-in; the default flow is "latest
  unstable."

For a solo dev or a permissive startup, fine. For a bank or a regulated
health org, this is a non-starter today.

### 3. Cognitive surface is larger than the docs admit

The tool juggles:

- bash 3.2 (macOS) + bash 5 (Linux) dialects with different style rules
- BSD sed + GNU sed
- Nix + native packages + a scope overlay layer
- Two opt-in prompt engines (oh-my-posh, starship) plus no-prompt default
- bash, zsh, PowerShell profiles all managed simultaneously
- A custom `nx` CLI, an `nx doctor` tool, a
  `nx_doctor.sh` file copied into `~/.config/nix-env/`
- A stray `scripts_egsave.ps1` at the repo root with no obvious purpose and
  not documented in `ARCHITECTURE.md`

Tests cover pieces of this, but onboarding a second maintainer would be
painful.

### 4. Scope creep vs. stated purpose

The README says "one command provisions a workstation." The repo delivers
that plus: overlay system, hook system, pin mechanism, MITM detection,
uninstaller, health checker, provenance recorder, profile migrator, CLI
subcommands, zsh/pwsh/bash coordinators, and two opt-in prompt engines.
Each is individually defensible in `ARCHITECTURE.md`, but collectively this
is a project that's hard to maintain sustainably by one person.

### 5. WSL path is untested end-to-end

`ARCHITECTURE.md:319` admits WSL orchestration isn't tested in CI (reasonable
justification given GitHub runner limits), but WSL is the #1 advertised use
case for Windows enterprise developers. "Validated by Pester unit tests that
mock `wsl.exe`" is not equivalent to integration testing. `wsl/wsl_setup.ps1`
is 755 lines of logic with no real-world CI gate.

### 6. Minor but telling code-quality issues

- `nix/setup.sh` disables SC2154 for an entire file
  (`# shellcheck disable=SC2154`) because globals cross phase boundaries.
  That's the architectural cost of the phase-library approach - globals are
  still globals, `# Reads/Writes` comments are not compiler-enforced.
- `_IR_*` + `_ir_*` + `$_scope_set` + `$any_scope` - the global namespace is
  dense and implicit.
- `_on_exit` reads six globals it doesn't declare (`setup.sh:54-67`) - if a
  phase renames one, the trap silently breaks.
- `CHANGELOG.md` has one entry: `[Unreleased]`. No versions, no releases.

## Concrete recommendations (if this is meant to grow up)

1. **Tone down the enterprise claims** in `docs/enterprise.md` and README.
   Replace "Strong" ratings with concrete evidence or drop to
   "Available / Partial / Missing." Current framing will burn credibility
   with the first serious reviewer.
2. **Ship v1.0 with signed tags and release tarballs** before any further
   enterprise pitch. The `VERSION` file fallback already anticipates this -
   finish the workflow.
3. **Default to a pinned nixpkgs SHA**, make `unstable` opt-in via
   `--unstable`. This inverts the current risk posture.
4. **Replace `curl | sh` in `_install_nix_darwin`** with a two-step
   download-then-verify-checksum flow, or refuse to auto-install and fail
   loudly with instructions.
5. **Either own the enterprise layer or stop advertising it.** Pick one:
   publish a reference telemetry collector + policy overlay in a companion
   repo, or reframe as "developer workstation bootstrapper with extension
   points."
6. **Delete `scripts_egsave.ps1`** or document it. Unexplained scripts at
   repo root erode trust.

## Who should actually use this, today

- **Yes**: individual developers on macOS/WSL who want a one-shot env setup,
  especially behind a corporate MITM proxy. The proxy handling alone is
  worth it.
- **Yes**: small platform teams (5-20 devs) willing to treat this as an
  internal fork.
- **Probably not yet**: regulated enterprises, air-gapped environments, or
  anything needing formal supply-chain controls - unless the organization
  is prepared to build the missing 40% (signing, telemetry, distribution,
  policy) themselves.

## Bottom line

The code is better than the marketing, and the marketing is better than the
actual enterprise posture. Trim the claims back to what the code delivers,
ship a real v1.0, harden the supply chain story, and this becomes a
genuinely credible tool in its niche. Today it's a strong personal/team
bootstrapper wearing a suit two sizes too big.
