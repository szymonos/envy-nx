## When to write a Codified-Learning trailer

Use the trailer when the change teaches something **general** that future contributors should know - a constraint, a non-obvious pattern, a class-of-bug to watch for. Skip it for one-off typos, dependency bumps, or trivial refactors.

Example commit body:

```text
fix(profile): guard prompt() against re-init under OMP

Codified-Learning: In pwsh profile scripts, never define `function Prompt`
unconditionally - the last function definition wins (case-insensitive global
function table), so a "fallback" prompt clobbers oh-my-posh's installed
prompt. Always guard with `if (-not $env:POSH_SHELL ...) { function Prompt {...} }`.
```

To opt out for a commit that genuinely yields no generalization (and that touches a high-leverage path the pre-commit hook watches), add `# no-learning` anywhere in the commit body.

## L-000 - 2026-05-23 - meta

**Source:** PR #N/A (seed entry) - Trigger: refactor
**Scope:** `design/lessons.md`, `.github/workflows/codify_learnings.yml`, `.pre-commit-config.yaml`

**What we learned:** Codified learnings need three properties to compound across the team: **tool-agnostic format** (so contributors using any AI agent or no agent at all can read and write), **git-tracked durability** (so they survive machine loss and propagate via the normal review flow), and **automatic capture** (so the discipline doesn't depend on contributor memory). The `Codified-Learning:` commit trailer + post-merge `codify_learnings.yml` workflow gives all three: trailers are plain text any tool can emit, the resulting `design/lessons.md` entries land on `main` via auto-merging PRs, and the workflow fires unconditionally on every merge.

**Why it matters:** Earlier attempts (`.wolf/cerebrum.md`) were Claude-only and gitignored - learnings stayed on individual machines and never reached teammates using other tools. The first contributor to hit the same gotcha re-discovered it from scratch.

**How to apply:** When a PR teaches a generalization, add a `Codified-Learning:` trailer in the commit body. After merge, watch the `codify` follow-up PR auto-create and auto-merge an entry into this file. See `CONTRIBUTING.md` § "Codifying learnings" for the full convention.

## L-001 - 2026-05-23 - render-heuristics

**Source:** PR #32 (merge SHA `e459dc2`) - Trigger: bug
**Scope:** `.assets/lib/nx_profile.sh`, `.assets/config/pwsh_cfg/profile_nix.ps1`

**What we learned:** Render-time heuristics that grep for bare substrings to detect "is this already handled" conflate *mention* with *mutation*. The `_has_local_bin` check in `nx_profile.sh` grepped for `.local/bin` anywhere in `~/.zshrc` outside managed blocks, intending to detect a user-added PATH entry. It matched lines that *referenced* the path without modifying PATH (`eval "$($HOME/.local/bin/uv ...)"`, copilot probes, comments) and suppressed the entire env block. The same shape appeared in bug-067: an unconditional `function Prompt` defined as a "fallback" silently clobbered oh-my-posh's installed prompt because PowerShell's global function table is last-writer-wins with no warning.

**Why it matters:** Both incidents shipped to users, passed tests (the tests had codified the buggy behavior as the expected contract), and presented with misleading symptoms. The substring-grep bug looked like "uv stopped working"; the prompt bug looked like "oh-my-posh loads the wrong theme." Debugging the symptom leads away from the root cause.

**How to apply:** When the rendered code's runtime can self-protect against duplication (e.g., `case ":$PATH:" in *":$dir:"*) ;;`), prefer that runtime guard over a render-time heuristic. If you must check at render time, match the actual mutation pattern (`PATH=.*\.local/bin`) not a bare substring. For function definitions, always guard with a probe for the prior init (`if (-not $env:POSH_SHELL ...) { function Prompt {...} }`).

## L-002 - 2026-05-23 - cache-staleness

**Source:** PR #36 (merge SHA `7340d49`) - Trigger: bug
**Scope:** `.assets/lib/nx.sh`, `nix/lib/phases/post_install.sh`

**What we learned:** Tools that cache rendered shell-init scripts (oh-my-posh, and potentially starship, fnm, direnv) can embed absolute `/nix/store/<hash>/bin/<tool>` paths in the cached output. The cache key is typically (config-hash, shell) - it does **not** include the binary version or store path. After `nix profile upgrade` + `nix store gc`, the old store path is garbage-collected but the cache still references it. Every prompt then fails with `bash: /nix/store/...: No such file or directory`. zsh's init cache does not embed the binary path, so zsh sessions stay healthy and mask the issue - making it look platform-specific when it is actually cache-specific.

**Why it matters:** The failure fires on every interactive prompt, not on an explicit command - users see a cryptic error on every keystroke with no obvious trigger. The same `ModuleAnalysisCache-*` / `StartupProfileData-*` staleness pattern affects pwsh. Any new tool that caches shell-init with embedded store paths will hit the same class of bug.

**How to apply:** After any operation that removes Nix store paths (`nix store gc`, `nix profile remove`), sweep shell-init caches that may embed those paths. The sweep lives in `_nx_clear_stale_caches` (called by `nx upgrade` / `nx gc`) and `_phase_post_install_clear_stale_caches` (called after `nix store gc` during setup). When adding a new tool with shell-init caching, add its cache glob to both sweep functions.
