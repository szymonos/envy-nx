## Principles preserved (do NOT compromise on these)

- Cross-platform: macOS, Linux, WSL, Coder/rootless
- 3 shells: bash, zsh, PowerShell
- 2 prompt engines: oh-my-posh + starship (justified in `docs/decisions.md`)
- Bootstrapper-disposable-repo model
- `install_atomic` + managed-block pattern

## Verdict upfront

The system is **more reliably structured than reliably operating** - invest in runtime telemetry (incremental provenance, behavior checks) over more static guards (parity hooks). The biggest cognitive-load tax isn't the prompt engines - it's the **5 simultaneous bash flavors**, **3 setup entry points**, and **4 reactive parity hooks** that exist because the manifest didn't go far enough.

## Reasoning per item

### 1. Generate from manifest, replace 4 parity hooks with 1 generation hook

**Problem.** §3d / §8 describe 4 hooks (`check-nx-completions`, `check-nx-profile-parity`, `check-nx-dispatch-parity`, `check-nx-lib-files-parity`) that parse code with regex to extract case arms / for-loops, then compare against `nx_surface.json`. Each was added reactively when drift bit. Generation is strictly more powerful than checking.

**Proposal.** Extend `tests/hooks/gen_nx_completions.py` to also emit:

- `nx_main`'s `case "$cmd" in` body (between `# >>> nx-main generated >>>` markers in `nx.sh`)
- `_aliases_nix.ps1`'s `switch ($subCmd)` body (between `#region nx:dispatch ... #endregion`)
- The lib-file list constants used by `phase_bootstrap_sync_env_dir`, `_nx_self_sync`, and `_check_env_dir_files` (between `# >>> nx-libs generated >>>` markers)

Then one hook (`check-generated-files-fresh`) replaces all four: rerun the generator, diff, fail if anything changed.

**Open questions.**

- Is there any case in `nx_main` that's _not_ a manifest-driven dispatch (e.g. argument preprocessing)? Need to scan before assuming pure routing.
- Lib-file list lives in 3 functions - generating a constant variable then referencing it from 3 places might require restructuring. Alternative: generate 3 separate code regions.

### 2. Re-evaluate bash 3.2 floor

**Problem.** §5 / §7.1 / §7.2 - every nix-path file lives under bash 3.2 + BSD constraints. This forces workarounds for `mapfile`, `declare -A`, `${var,,}`, negative indices, BSD sed. The justification is "stock macOS ships bash 3.2". But macOS 14+ ships zsh as default; bash 3.2 only matters if a user runs `bash setup.sh` on un-brewed macOS _before_ installing nix.

**Proposal.** Audit:

- How many users actually hit the bash-3.2 path? (CI is mostly Linux + Determinate macOS.)
- If we drop 3.2, what's the floor? bash 4 (still common) or bash 5 (modern)?
- Migration impact: `scopes.sh` (~40% workarounds), `install_record.sh`, `nx_*.sh` family files. Probably 100-200 lines deleted.

If we drop: keep zsh-compat constraints (those are about zsh sourcing, not bash version); update `check-bash32` → `check-bash4plus`; relax shellcheck.

**Open questions.**

- Is there a real `bash setup.sh` on stock macOS use case, or is it `zsh setup.sh` everywhere?
- Determinate Nix installer puts modern bash on PATH after install - does `setup.sh` re-exec under it after bootstrap? If yes, the bash-3.2 window is just the first 50 lines.

### 3. Write `install.json` incrementally per phase

**Problem.** §3a / §13.6 - `install.json` is written by an EXIT trap. Doesn't fire on SIGKILL, OOM, kernel panic, or when a child process overrides the trap. The case where we most need provenance ("what phase died?") is the case we're most likely to miss.

**Proposal.** Replace single EXIT write with per-phase incremental writes via a new `_ir_flush <phase> <status>` helper in `install_record.sh`:

- `bootstrap` → write immediately on entry (status=in_progress)
- `bootstrap` complete → flush (status=success, phase=platform)
- `platform` → flush (status=in_progress)
- ...
- final → flush (status=success, phase=complete)

Atomic write per call (temp + rename). The EXIT trap stays as a safety net for unexpected exits - flushes whatever the current `_ir_phase` was as `status=interrupted`.

**Open questions.**

- Per-write cost: ~6-8 phases × 1 write = 8 atomic writes. Negligible.
- Backwards compat for `install.json` schema: add `phase_history[]` field or just overwrite each time?

### 4. `exec` instead of exit-and-rerun on auto-refresh

**Problem.** §3a - `phase_bootstrap_refresh_repo` exits 0 with "run the script again". Surprising for humans, bug magnet for CI scripts that don't expect setup.sh to "succeed without doing anything."

**Proposal.** After successful pull, `exec "$0" "$@"` (with `NX_REEXECED=1` guard to prevent loops). User experience: "ran once, magic happened." CI experience: same.

**Open questions.**

- What if HEAD changed but the script itself wasn't part of the change? Re-exec is harmless; the cached parsing just re-runs.
- Detached-HEAD or worktree edge cases - what does `git pull` do? Already gated by §3a's "Skips silently when..." conditions; should be fine.

### 5. Document the GC-vs-rollback trade-off

**Problem (future-self framing).** `phase_post_install_gc` runs `nix profile wipe-history` + `nix store gc` on every successful setup, which means `nx rollback` only works _within_ a single setup invocation, not _across_ completed runs. An external reviewer (or future me) seeing this in code with no rationale captured will keep flagging it as a reliability weakness - exactly what just happened in this review. The choice is intentional but undocumented.

**Why it's intentional (per maintainer).**

- **WSL VHD growth is a one-way ratchet.** The virtual disk grows on demand and never auto-shrinks. Reclaiming space requires `wsl --shutdown` + `diskpart` + `compact vdisk` - ceremony users won't perform. Letting nix-env generations accumulate would balloon the VHD permanently within weeks of normal `nx upgrade` use.
- **Maintainer-trust assumption.** `nix/setup.sh` and `nx setup` runs are tested released versions. The recovery path for a bad release is "re-run the prior release", not "rollback to a previous generation on disk".
- **Rollback within a run still works.** The current generation is preserved up to the next setup completion, so rollback during/after a single setup run remains intact.

**Proposal.**

1. Add a `### Why nix gc runs in post-install by default` entry to `docs/decisions.md` (between _Why bash 3.2 compatibility_ and _Why oh-my-posh and starship_ feels like the right location). Cover the WSL VHD ratchet, the maintainer-trust assumption, and the explicit trade-off (cross-run rollback is sacrificed for bounded store size).
2. Update the ARCHITECTURE.md §3a post-install dispatch table line `Common post-install + nix garbage collection` → add a brief "(bounded store; see decisions.md)" pointer so the rationale is one cross-link away when reading the architecture doc.
3. Optionally, update `nx doctor`'s `install_record` or `nix_profile` Fix hint to mention "previous generation has been GCd by design - re-run prior release if needed" so users hitting the rollback gap get a clear diagnosis.

**Open questions.**

- Should we also add a CHANGELOG entry framing this as "documenting an intentional choice"? Probably not - it's a doc clarification, not a behavior change.
- Worth a `--keep-history` opt-out flag for advanced users on non-WSL platforms (macOS/Coder) where VHD growth isn't a concern? Defer unless requested - adds surface for marginal value.

### 6. Behavior checks in `nx doctor`

**Problem.** §3c - all 13 checks audit shape, not behavior. `env_dir_files` checks file _existence_; doesn't verify content matches `.assets/lib/`. `shell_profile` counts blocks; doesn't verify the block is what `nx profile regenerate` would produce. So all 13 checks can pass on a broken install.

**Proposal.** Add 1-2 behavior checks:

- `managed_block_drift`: regenerate the `nix-env managed` block to /tmp via `_nx_render_nix_block`, diff against current `~/.bashrc` block. FAIL if differ. Catches manual edits and version skew.
- `scope_bins_in_profile`: for each scope in `config.nix`, verify its `# bins:` resolve under `~/.nix-profile/bin/` specifically (not just PATH). Tighter than current `scope_binaries`.

**Open questions.**

- `managed_block_drift` would FAIL after a `nx self sync` if the user hadn't yet rerun `nx profile regenerate`. Is that the right UX or noise?
- Performance: regenerating a block is ~ms, fine. Stat'ing 50 binaries is also fine.

### 7. Validate MITM probe failure cause

**Problem.** §3g - `phase_nix_profile_mitm_probe` probes a TLS endpoint and runs `cert_intercept` on failure. There's no validation that the failure was _cert-related_. DNS failure / captive portal / transient block all trigger cert extraction; whatever the probe returns gets appended to `ca-custom.crt` and survives.

**Proposal.** After probe failure, run a second probe with `curl --cacert /dev/null` (forces no cert validation). If that also fails, it's not a cert problem - log "TLS probe failed for non-cert reason" and skip cert extraction. If it succeeds, continue with current logic.

**Open questions.**

- `--cacert /dev/null` semantics: curl might still use system CAs. May need `-k` (insecure) instead. Need to test.
- What about partially-failing networks where `google.com` probe fails but other endpoints work? Edge case; probably not worth optimizing for.

### 8. `nix` version floor check

**Problem.** §3c - `_check_nix_available` checks `command -v nix`. Doesn't catch nix < 2.4 (no flake support → cryptic errors).

**Proposal.** Add `nix --version` parse + version compare. Fail if < 2.18 (current nixpkgs-unstable floor). 5 lines.

### 9. Structured configure-script error reporting

**Problem.** §5 - wrapper boundary at the phase level means configure scripts (`gh.sh`, `git.sh`, `conda.sh`, etc.) call `gh`/`git`/`curl` directly. When `configure/conda.sh` dies, you get conda's error stream, not "configure_conda failed at miniforge install." `_io_run` captures the _script's_ failure but not the internal step.

**Proposal.** Add an `_io_step "<label>"` helper sourced into configure scripts that prints `>>> step: install miniforge` before each major action. On failure, the last `_io_step` label is included in the error log.

**Open questions.**

- Adds noise to successful runs. Maybe only print on failure (capture last label, surface in `_io_run`'s error path).
- Worth doing if we also do #3 (incremental `install.json`)? Yes - different layers (per-phase vs per-step within configure).

### 10. Collapse `linux_setup.sh` into `setup.sh --system-prep`

**Problem.** §1 / §3a - three setup entry points (`nix/setup.sh`, `linux_setup.sh`, `wsl/wsl_setup.ps1`). The Linux system-prep work (apt/dnf upgrade, base packages, nix bootstrap) could be a phase in `setup.sh`.

**Proposal.** Add `phase_system_prep_*` that activates when `[ $EUID -eq 0 ]` and `/etc/os-release` is present. `linux_setup.sh` becomes a thin wrapper or disappears. WSL stays - it has to run on Windows.

**Open questions.**

- Does running `setup.sh` as root with `--system-prep` then dropping to user via `sudo -u` work cleanly? `nix/setup.sh` currently rejects root.
- Is the 2-script split worth keeping for "system stuff" vs "user stuff" mental separation? Maybe split phases visually but keep one entry point.

### 11. Rename managed blocks (eliminate word-order confusion)

**Problem.** §3e - `# >>> nix-env managed >>>` and `# >>> managed env >>>` differ only in word order. Easy to typo, easy to confuse.

**Proposal.** Rename to `# >>> nix:managed >>>` and `# >>> env:managed >>>` - matches the PowerShell `#region nix:*` convention already in use. One-time migration: `nx profile regenerate` rewrites both block names; uninstaller handles both old + new names for one release.

**Open questions.**

- Migration: leave a one-shot migrator in `nx profile regenerate` to convert old → new names? Run once, log "migrated managed-block names."
- CHANGELOG breaking-change call-out - minor, but document.

### 12. Address the `flake.lock` reproducibility gap

**Problem.** §5 - no repo-level `flake.lock`. Per-user lock means two devs on the same team get different lockfiles. `nx pin` is opt-in. We market reproducibility but deliver "your machine, today."

**Proposal.** Either:

- (a) Ship `nix/recommended.lock` users can opt into via `nx pin set --recommended`. Updated periodically (monthly?) by CI.
- (b) Update docs (`docs/decisions.md`, README) to call out clearly: "reproducibility on one machine; cross-machine reproducibility requires `nx pin`."

(b) is cheap and honest. (a) is a step toward fleet management.

## Don't touch

These are load-bearing and well-executed:

- Atomic file install (`install_atomic`) - race protection is real
- Managed-block pattern - full regeneration > append, correct call
- 2 prompt engines - small delta (~2-3% codebase), real user value, justified
- Bootstrapper-disposable-repo model - genuinely good
- Side-effect wrappers (`_io_*`) at phase level - testable, right boundary
- Manifest-driven completions - extending it (item #1) is the win, not replacing it

### Block E - Bigger changes ✅ both resolved-not-changed

1. **#2 Bash 3.2 audit** ✅ resolved-not-changed. After alignment: the constraint stays. Cost (~150 lines of workarounds) is real but bounded; benefit is zero given the install base. Real-world counter-evidence: first user (senior devops) hit bash 3.2 on stock macOS Terminal immediately, before any compat work landed - the population that "must work always" is exactly those users. Ships under "Block F" instead: invest in tooling so the constraint is testable. (a) Force `/bin/bash` explicitly in `.github/workflows/test_macos.yml` so CI tests against Apple's 3.2 regardless of PATH state. (b) Capture `bash_version` in `install.json` provenance + show in `nx version` so bug reports surface "is this a bash 3.2 path failure?" automatically.
2. **#10 Collapse `linux_setup.sh` into `setup.sh --system-prep`** ✅ resolved-not-changed. The split is intentional and load-bearing for the **root vs user-scope boundary**: `nix/setup.sh` rejects root by design (its operational contract is "I touch only your home"), while `linux_setup.sh` orchestrates the parts that need `sudo` (base packages, nix bootstrap, root-required scope installers). Combining would either (a) violate the user-scope contract by adding `sudo` calls inside `nix/setup.sh`, or (b) create a single script with two fundamentally different execution models depending on a `--system-prep` flag. Maintenance cost of `linux_setup.sh` is bounded (~150 lines of mostly mechanical delegation, edits ~2x/year). Real-world usage is rare (most users come via WSL or directly to `nix/setup.sh` on macOS/Coder). Captured in `docs/decisions.md` as "Why two Linux entry points".

### Block F - Tooling for #2 ✅ shipped

1. **macOS CI uses `/bin/bash` explicitly** - workflow-wide `defaults.run.shell: /bin/bash -e {0}` + every script invocation prefixed `/bin/bash` so the shebang's `env bash` PATH lookup can't accidentally pick up brewed bash 5. Added a fail-fast version-check step at workflow start that asserts `/bin/bash` is bash 3.x.
2. **`bash_version` in install.json** - one new field captured from `${BASH_VERSINFO[0]}.${BASH_VERSINFO[1]}` (always available in any bash-executed script). Surfaces in `nx version` as `Bash: 3.2`. Lets future bug-report triage answer "is this a bash 3.2 path?" without asking.
3. **Drive-by fix**: macOS workflow's marker assertions still referenced legacy `nix-env managed` from before #11 - updated to the new `nix:managed` / `env:managed` names.
